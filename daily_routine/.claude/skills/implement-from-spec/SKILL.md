# 設計書に基づく実装スキル

設計書（`/docs/designs/`）に記載された実装計画に従い、コードを実装する。

## 前提知識

このスキルを実行する前に、以下のファイルを必ず読み込むこと:

1. **ガイドライン**: [/docs/guidelines/development_process.md](/docs/guidelines/development_process.md) — 開発プロセスとドキュメント体系を把握する
2. **対象設計書**: `/docs/designs/` 内の指定された設計書
3. **仕様書**: `/docs/specs/` 内のファイル — プロジェクト全体のビジョンと設計書の位置づけを確認する
4. **既存ADR**: `/docs/adrs/` 内のファイル — 採用済み技術の制約を確認する（存在する場合）
5. **既存の設計書**: `/docs/designs/` 内の他の設計書 — 依存関係や共通スキーマを確認する

## 入力

$ARGUMENTS

**必須:** 設計書の `.md` ファイル名を指定すること。ファイル名が指定されていない場合はエラーとし、`/docs/designs/` 内の設計書一覧を提示してユーザーに選択を促す。

引数の形式:

- `{設計書ファイル名}.md` — 設計書の全ステップを実装する（例: `project_skeleton_design.md`）
- `{設計書ファイル名}.md ステップN` — 指定ステップのみ実装する（例: `project_skeleton_design.md ステップ2`）

## 実行手順

### 1. 入力の検証と設計書の読み込み

1. 引数に `.md` ファイル名が含まれているか検証する
   - **ファイル名がない場合**: `/docs/designs/` 内の設計書一覧を取得し、ユーザーに `AskUserQuestion` で選択を促す。選択されるまで実装に進まない
   - **ファイル名がある場合**: `/docs/designs/{ファイル名}` を読み込む。ファイルが存在しない場合はエラーを報告する
2. 設計書の全セクションを読み込み、以下を把握する:
   - **スコープ**: 対象範囲と対象外
   - **技術設計**: アーキテクチャ、データ構造、インターフェース仕様
   - **実装計画**: 各ステップの内容と完了条件
   - **テスト方針**: 検証方法

### 2. 依存関係の確認

1. 設計書が参照する他の設計書・仕様書を確認する
2. 他の設計書で定義された共通スキーマやインターフェースがある場合、その実装が既に存在するか確認する
3. 未実装の依存がある場合、ユーザーに報告し判断を仰ぐ

### 3. 現在の実装状況の確認

1. 設計書で定義されたディレクトリ構造・ファイルが既に存在するか確認する
2. 既存コードがある場合、設計書との差分を把握する
3. 設計書の実装計画のうち、どのステップまで完了しているかを判断する

### 4. ステップごとの実装

設計書の「実装計画」セクションに定義された各ステップを順に実行する。

各ステップで以下を行う:

1. **実装開始前の確認**: ステップの内容と完了条件をユーザーに提示し、開始確認を取る
2. **コードの実装**: 設計書の技術設計セクションに記載されたコード・仕様に忠実に実装する
3. **完了条件の検証**: 設計書に記載された完了条件を実行・確認する
4. **ステップ完了の報告**: 結果をユーザーに報告する

### 5. テストの実施

設計書の「テスト方針」セクションに従い、テストを作成・実行する:

1. 設計書で定義されたテスト項目を実装する
2. テストを実行し、結果を確認する
3. 失敗がある場合は修正する

### 6. 実装結果のフィードバック

ガイドラインに従い、実装を通じて得られた知見を報告する:

1. **設計書との乖離**: 実装中に設計書の内容と異なる判断をした箇所があれば報告する
2. **仕様書へのフィードバック**: 実装を通じて仕様書に反映すべき知見があれば報告する
3. **未決事項の解決**: 設計書の「未決事項」セクションに記載された項目について、実装を通じて判断が得られた場合は報告する

## 実装時の原則

### 設計書に忠実に実装する

- 設計書に記載されたコード例・スキーマ定義・インターフェースは、そのまま実装の基礎とする
- 設計書に明記されていない実装判断が必要な場合は、`AskUserQuestion` でユーザーに確認する
- 設計書の「対象外」に記載された内容は実装しない

### ADRで採用済みの技術を尊重する

- `/docs/adrs/` に記録された技術選定に従う
- ADRと設計書の間に矛盾がある場合、ADRを優先し、ユーザーに報告する

### 既存コードとの整合性を保つ

- 他の設計書で定義・実装済みの共通スキーマやインターフェースを再利用する
- 既存のコーディングスタイル・命名規則に合わせる
- パッケージ構成は `project_skeleton_design.md` で定義された構造に従う（存在する場合）

### 段階的に進める

- 一度に大量のコードを生成せず、ステップごとに実装・確認を繰り返す
- 各ステップの完了条件を満たしてから次のステップに進む
- エラーが発生した場合はその場で対処し、原因と対処をユーザーに報告する

## 注意事項

- 設計書の「対象外」に含まれる機能は実装しない
- 新たな技術選定が必要な場合は実装を止め、ADR作成の必要性をユーザーに報告する
- 設計書に記載のない大きな設計判断を独断で行わない
- テスト方針が設計書に記載されている場合は必ずテストを作成する
