# 設計書作成スキル

仕様書（`/docs/specs/initial.md`）に基づき、指定されたサブタスクの設計書を `/docs/designs/` に作成する。

## 前提知識

このスキルを実行する前に、以下のファイルを必ず読み込むこと:

- プロジェクトガイドライン: [/docs/guidelines/development_process.md](/docs/guidelines/development_process.md)
- 現在の仕様書: [/docs/specs/initial.md](/docs/specs/initial.md)
- 既存の設計書: `/docs/designs/` 内のファイル一覧
- 既存のADR: `/docs/adrs/` 内のファイル一覧

## 入力

$ARGUMENTS

## 実行手順

### 1. コンテキストの把握

1. `/docs/guidelines/development_process.md` を読み、ドキュメント体系と開発プロセスを確認する
2. `/docs/specs/initial.md` を読み、プロジェクト全体の仕様を把握する
3. `/docs/designs/` 内の既存設計書を確認し、重複や依存関係を把握する
4. `/docs/adrs/` 内の既存ADRを確認し、既に決定済みの技術選定を把握する

### 2. サブタスクの特定

- 引数で指定されたサブタスクが仕様書のどの部分に対応するかを特定する
- 対象範囲（スコープ）を明確にする

### 3. ユーザーとの対話による設計の精緻化

- `AskUserQuestion` を使い、設計に必要な判断をユーザーに確認しながら進める
- 以下のような観点で質問する:
  - 技術選定の候補がある場合、どれを採用するか
  - スコープの境界が曖昧な場合、どこまでを対象とするか
  - 実装上のトレードオフがある場合、どちらを優先するか

### 4. 設計書の作成

- ファイル名: `/docs/designs/{サブタスクを端的に表す英語名}_design.md`
- 以下の構成で記載する:

```markdown
# {サブタスクのタイトル} 設計書

## 1. 概要

- 対応する仕様書セクション
- このサブタスクで実現すること

## 2. スコープ

- 対象範囲
- 対象外（明示的に除外するもの）

## 3. 技術設計

- アーキテクチャ / データフロー
- 使用技術・ライブラリ
- データ構造 / インターフェース仕様

## 4. 実装計画

- 実装ステップ（順序付き）
- 各ステップの完了条件

## 5. テスト方針

- どのように動作を検証するか

## 6. 未決事項

- 実装を進める中で判断が必要な項目
```

### 5. 技術選定が発生した場合

- 新しい技術・ライブラリ・サービスの採用が必要な場合は、ADRも作成する
- ファイル名: `/docs/adrs/{連番}_{選定対象の概要}.md`
- 連番は既存ADRの最大番号 + 1 とする
- ガイドラインに定められたADRの記載内容に従う

### 6. 仕様書の更新確認

- 設計の過程で仕様書に不足・矛盾が見つかった場合、ユーザーに報告する
- ユーザーの承認を得た上で `/docs/specs/initial.md` を更新する(スキル `/docs-update-spec` を呼び出す)
