# layout-research

間取りタイプに応じたレイアウト事例をWeb検索で調査し、設計原則を抽出するスキル。
調査結果をレポートにまとめ、デザインスコアリング基準（`scoring_criteria.json`）を生成する。

## 前提条件

- `floor_plan_complete.svg` と `room_info.json` が存在すること（間取り情報の参照に必要）
- `assets.json` が存在すること（家具情報の参照に必要）

## 入力

- `floor_plan_complete.svg` — 間取り完成版SVG（画像として読み空間を把握）
- `room_info.json` — 部屋・設備・配置不可領域の座標データ
- `assets.json` — 家具アセット情報

## 出力

- `layout_design_principles.md` — リファレンス調査レポート（画像参照付き）
- `research_images/` — ダウンロードした参考画像
- `scoring_criteria.json` — デザインスコアリング基準（標準+ユーザー要望）

---

## 実行手順

### Step 1: 空間と家具の把握

`floor_plan_complete.svg` を画像として読み、`room_info.json` と `assets.json` を読んで以下を把握:
- 部屋タイプ（1K/1LDK/2LDK等）、各部屋のサイズ
- 配置する家具の種類と数
- 固定設備の位置（キッチン、ベランダ等）

### Step 2: 調査条件の確認

ユーザーに以下を質問し、調査条件を確定する:

1. **居住者情報**: 何人暮らし？在宅ワーク？ペット？
2. **用途の特徴**: 在宅ワーク重視？リラックス重視？
3. **特に重視したい点**: あれば（例: 「デスクが向かい合わないようにしたい」）
4. **参考にしたいスタイル**: あれば（例: 「ミニマリスト」「北欧風」）

### Step 3: Web検索・事例収集・画像取得（統合）

調査条件に基づいてWeb検索・Webページ閲覧を行う。**任意のWeb検索およびWebコンテンツのFetchが可能**。

検索キーワード例:
- `{間取りタイプ} {用途} レイアウト 実例`
- `{間取りタイプ} {人数}人暮らし 家具配置`
- `{間取りタイプ} ワークスペース インテリア`
- `small apartment layout {furniture type} tips`

検索の進め方:
1. 日本語・英語の両方で検索する
2. 最低5件の事例ページを閲覧する

各事例ページについて、以下を **1ページずつ完結** させる:

1. WebFetch でページ内容を読み取り、配置パターンと設計意図を抽出
2. **同じ WebFetch で** ページ内の画像URL（img src）も抽出する
3. 抽出した画像URLから有用なもの（間取り図・配置図解・レイアウト写真）を選別
4. 即座に curl でダウンロード
5. ダウンロードした画像を Read で確認し、レイアウト原則の根拠になるか判定
   - 人物写真、商品写真、ロゴ等は除外
6. 有用な画像のファイル名に事例番号を含める

```bash
mkdir -p {output_dir}/research_images
curl -L -o {output_dir}/research_images/ref_01a_parallel_layout.jpg "{画像URL}"
```

画像ファイル名: `ref_{事例番号}{連番}_{内容}.{拡張子}`
- 例: `ref_01a_parallel_layout.jpg`（事例1の1枚目、Ⅱ型レイアウト図）
- 例: `ref_02a_window_desk.jpg`（事例2の1枚目、窓際デスク配置）

**禁止事項**:
- 事例ページに載っていない別サイトから画像を探して代用しない
- レポートに記載したURLと画像の出典が一致していること

### Step 4: 画像品質チェック

ダウンロードした全画像を Read で閲覧し、以下を確認:
- [ ] レポートの事例URLと画像の出典が一致している
- [ ] 画像がレイアウト原則の根拠として有用（間取り図・配置図解・実例写真）
- [ ] 人物写真・商品写真・ロゴ・広告画像は除外済み
- [ ] 各事例に最低1枚の画像が紐づいている

不適切な画像は削除し、必要に応じて事例ページから再取得する。

### Step 5: 設計原則の抽出

事例から共通する設計原則を抽出する。各原則には:
- **原則名**: 短い名前（例: 壁面密着、採光優先度）
- **説明**: 何を意味するか
- **根拠**: どの事例からどう導かれたか
- **スコアリング変換案**: どう定量化できるか

### Step 6: レポートの作成

`{output_dir}/layout_design_principles.md` を以下の構造で作成:

```markdown
# レイアウトデザイン原則: {間取りタイプ} {用途}

## 調査条件
- 間取り: {タイプ}, リビング約{面積}㎡
- 居住者: {人数}人, {用途}
- 検索キーワード: ...

## 調査した事例

### 事例1: {タイトル}
- URL: {URL}
- 参考画像:
  - ![説明1](research_images/ref_01a_xxx.jpg) — {何を示す画像か}
  - ![説明2](research_images/ref_01b_xxx.jpg) — {何を示す画像か}
- （画像なしの場合）画像取得不可の理由: {403エラー、画像が動的ロード等}
- 要約: {この事例の配置パターンと特徴}
- 抽出した原則: {関連する原則名}

### 事例2: {タイトル}
...

## 設計原則一覧

### 原則1: {原則名}
- 説明: {何を意味するか}
- 根拠: {事例からの根拠}
- スコアリング変換案: {定量化の方法}

### 原則2: {原則名}
...

## スコアリング基準案
（scoring_criteria.json の素案をここにまとめる）
```

### Step 7: ユーザー要望の収集

レポートを提示した後、ユーザーに質問する:

> レポートの設計原則を確認しました。
> これらに加えて、**特に重視したい観点**はありますか？
> 例: 「デスク間の距離を十分に取りたい」「ダイニングは窓際にしたい」

ユーザーの回答を `custom` 観点として記録する。

### Step 8: scoring_criteria.json の生成

レポートの原則（standard）+ ユーザー要望（custom）を合成して `scoring_criteria.json` を生成:

```json
{
  "source": "layout_design_principles.md",
  "criteria": {
    "standard": [
      {
        "id": "{原則ID}",
        "label": "{原則名}",
        "description": "{説明}",
        "weight": 1.0
      }
    ],
    "custom": [
      {
        "id": "{ユーザー要望ID}",
        "label": "{要望名}",
        "description": "{説明}",
        "weight": 0.9
      }
    ]
  }
}
```

**weight の目安**:
- 1.0: 最重要（壁面密着、中央開放など基本原則）
- 0.8: 重要（採光優先度、機能グルーピングなど）
- 0.7: 考慮すべき（好みに依存するもの）
- 0.9: ユーザー要望（ユーザーが明示的に重視したもの）

### Step 9: 確認

以下をユーザーに提示し、確認を取る:
1. レポートの内容
2. `scoring_criteria.json` の観点一覧と重み
3. 修正があれば反映する
