# 設計書オーケストレーション

並列・独立で作成された設計書（`/docs/designs/`）間の技術的整合性を検証し、乖離を検出・修正提案するスキルです。

## 説明

設計書は仕様書のサブタスクごとに独立して作成されるため、以下のような乖離が発生し得ます：

- ある設計書で定義したスキーマ・インターフェースを、別の設計書が異なる形で前提としている
- ディレクトリ構造やファイル配置規約が設計書間で統一されていない
- 技術スタックの選定（ライブラリ・フレームワーク）が矛盾している
- 仕様書で定義された制約に対して設計書が準拠していない

このスキルは上記のような整合性の問題を体系的に検出します。

## 使用方法

```bash
/design-orchestration
```

パラメータは不要です。

## チェック対象

### 対象に含むドキュメント

- 設計書: `/docs/designs/` 直下の `*_design.md` ファイル
- 仕様書: `/docs/specs/initial.md`
- ガイドライン: `/docs/guidelines/development_process.md`
- ADR: `/docs/adrs/` 配下のファイル（採用済み技術の制約として参照）
- サブタスク分解計画: `/docs/claude-plans/` 配下のファイル（依存関係・スコープの参照）

### 対象外

- `/docs/designs/archives/` 以下のファイル（開発完了済み）

## チェック項目

### 1. スキーマ・データ構造の整合性

プロジェクト骨格設計書で定義された共通スキーマ（Pydanticモデル）が、他の設計書で参照・利用される箇所と一致しているかを検証する。

- フィールド名・型が一致しているか
- 入出力データの受け渡しに矛盾がないか（あるモジュールの出力型が次のモジュールの入力型と整合するか）
- 新たに定義されたデータモデルが共通スキーマと重複・矛盾していないか

### 2. ディレクトリ構造の整合性

各設計書で定義されるソースコード配置・ランタイムデータ配置が相互に矛盾しないかを検証する。

- ソースコードのモジュール配置パス（例: `src/daily_routine/evaluation/` vs `poc/`）に統一された規約があるか
- ランタイムデータの配置パス（例: `{data_root}/evaluations/` vs `poc/video_ai/`）に統一された規約があるか
- 設計書間で同じディレクトリを異なる用途で使おうとしていないか

### 3. 技術スタックの整合性

全設計書で採用・前提としている技術が統一されているかを検証する。

- Python バージョン、パッケージマネージャ、主要ライブラリのバージョンが一致しているか
- 同一目的に対して異なるライブラリを選定していないか
- ADRで採用決定済みの技術と矛盾する選定をしていないか
- 同期/非同期の方針（例: `async def` vs 同期関数）が統一されているか

### 4. インターフェース・契約の整合性

モジュール間のインターフェース（関数シグネチャ、クラス定義、プロトコル）が整合しているかを検証する。

- あるモジュールが公開するインターフェースと、それを利用する側の設計書が想定するインターフェースが一致しているか
- 共通クライアントインターフェース（例: `ImageGeneratorClient`）のメソッドシグネチャが一貫しているか
- エラーハンドリングの方針が統一されているか

### 5. 仕様書との整合性

各設計書の内容が仕様書（`/docs/specs/initial.md`）と矛盾していないかを検証する。

- 設計書で扱うスコープが仕様書の該当セクションとずれていないか
- 仕様書で定義されたデータ項目が設計書のスキーマで漏れなくカバーされているか
- 仕様書の出力仕様（解像度、フレームレート等）が設計書で正しく反映されているか

### 6. ガイドラインへの準拠

各設計書がプロジェクトのガイドライン（`/docs/guidelines/development_process.md`）に従っているかを検証する。

- 設計書の構成（セクション構造）がガイドラインで定められた形式に従っているか
- ADRの作成が必要な技術選定が行われている場合、ADRの作成計画が含まれているか
- 仕様書へのフィードバック（不足・矛盾の検出）が設計書に記録されているか

### 7. サブタスク間の依存関係・スコープの整合性

サブタスク分解計画で定義された依存関係とスコープが、各設計書の「スコープ」「対象外」セクションと整合しているかを検証する。

- 設計書のスコープが分解計画のサブタスク定義と一致しているか
- 設計書の「対象外」が依存先の設計書でカバーされているか
- 依存関係のある設計書間で前提条件に矛盾がないか

## 実行手順

1. **対象ドキュメントの収集**
   - `/docs/designs/` 直下の `*_design.md` ファイル一覧を取得（`archives/` は除外）
   - `/docs/specs/initial.md` を読み込む
   - `/docs/guidelines/development_process.md` を読み込む
   - `/docs/adrs/` のファイルを読み込む（存在する場合）
   - `/docs/claude-plans/` のファイルを読み込む

2. **全ドキュメントの読み込みと構造化理解**
   - 各設計書の主要セクション（スキーマ定義、ディレクトリ構造、技術スタック、インターフェース、スコープ）を抽出・理解する

3. **チェック項目の順次検証**
   - 上記7つのチェック項目を順に検証する
   - 各項目について、問題がなければ「OK」、問題があれば具体的な乖離内容を記録する

4. **結果レポートの出力**
   - 検出された問題を以下の形式でまとめて報告する：

   ```
   ## 整合性チェック結果

   ### [チェック項目名]
   **ステータス:** OK / 要修正

   #### 問題 N: [問題の要約]
   - **関連設計書:** [ファイル名]
   - **乖離内容:** [具体的な不整合の説明]
   - **修正提案:** [どう修正すべきか]
   - **影響範囲:** [修正が影響する他のドキュメント]
   ```

5. **修正の実行**
   - ユーザーに修正提案を提示し、承認を得た上で修正を実行する
   - 修正は1つずつ確認しながら進める
   - 仕様書の更新が必要な場合、その旨をユーザーに報告する

## 注意事項

- このスキルは設計段階の並列作成による乖離を早期発見することを主目的とする
- 修正はユーザーの承認後に実行する。自動修正は行わない
- 設計書の技術的判断（どちらの設計が正しいか）が必要な場合は、ユーザーに判断を委ねる
- ADRで採用決定済みの技術は「正」として扱い、それに矛盾する設計書側を修正対象とする
- 仕様書とのずれが検出された場合、設計書側の修正が適切か仕様書側の更新が適切かをユーザーに確認する
