# PoC: SVG間取り図 → Blender 3Dモデル変換

## 概要

`floor-plan-filter`スキルで生成した壁線SVG（`_walls.svg`）をBlenderに取り込み、3D壁モデルを生成する。

## 検証済みの内容

### SVGインポート → 3D化（完了）

`svg_to_3d.py` で以下を実現：

- `bpy.ops.import_curve.svg()` でSVGをBlenderにインポート
- 2DカーブのextrudeでZ方向に押し出し（壁高さ2.5m）
- スケール自動補正（長辺=10mと仮定）
- コレクション（壁・床）の自動整理
- `.blend` ファイルとして保存

**結果**: 3Dモデルの生成自体は成功。寸法も妥当（4.5m × 10.0m × 2.5m）。

### 要素分離の試行（課題あり）

Blenderの `Separate by Loose Parts` で1つの複合メッシュを39個のオブジェクトに分離したが、以下の問題がある：

- **壁・ドア弧・設備が正しく分類できない** — BBoxの面積・アスペクト比だけでは区別不可能
- **物理的に接続された要素は分離できない** — 壁が角やT字接合で繋がっているため、fillベースの表現では個々の壁セグメントに分割不可

## SVG要素分離の経緯

### 問題の本質

potraceが生成するSVGは**塗り潰し（fill）ベース**の表現：

```
壁 = 黒い塗り潰し領域（正の空間）
部屋 = 塗り潰し内の穴（負の空間）
```

この表現では：

- 外壁・間仕切り壁が**1つの連続した塗り潰し領域**を形成
- 壁に接触するドア弧・設備もfillを共有して接続
- Loose Partsで分離しても、fillで繋がった要素は1オブジェクトのまま残る

### 試行した画像処理アプローチ

| アプローチ             | 手法                              | 結果           | 問題                                   |
| ---------------------- | --------------------------------- | -------------- | -------------------------------------- |
| fill → Loose Parts     | Blenderメッシュ分離               | 39オブジェクト | 主構造が1〜3個の大オブジェクトに留まる |
| スケルトン化 → potrace | ImageMagick Thinning → 中心線抽出 | 3パス          | 壁が交差点で繋がったまま               |
| エッジ検出 → potrace   | ImageMagick Edge → 輪郭線抽出     | 38パス         | 主構造の輪郭はやはり1〜2パスに集約     |

**結論**: 画像処理だけでは、物理的に接続された壁セグメントの分離は困難。交差点を検出してグラフ分解するか、別のアプローチが必要。

### BBoxベースの分類の限界

Loose Partsで分離した39オブジェクトに対し、BBox（幅・奥行・面積・アスペクト比）で分類を試みた：

```
面積 ≥ 0.5㎡ → 壁
面積 < 0.5㎡ & AR ≤ 2.0 → 柱
それ以外 → その他
```

**問題点**:

- 薄い壁セグメント（min辺 < 0.06m、高AR）が「その他」に分類される
- ドア弧（0.3〜0.7m四方）と設備（浴槽・トイレ）がどちらも「柱」に分類される
- 形状の違い（曲線 vs 矩形）をBBoxでは判別できない

## 方針: Claudeによる視覚的分類

### アプローチ

画像処理やBBoxヒューリスティックではなく、**Claudeの画像認識能力**で各要素を分類する。

### Claudeに渡すデータ

分類はSVGレベルで行う。Blenderは最終インポートのみ。

#### 1. 画像データ

| 画像 | ファイル | 目的 |
|---|---|---|
| **アノテーション付きSVG画像** | `output/{stem}_indexed_annotated.png` | `floor-plan-filter` Step 1で生成済み。各パスに番号ラベルが付いており、形状・位置を視認できる |
| **元の間取り図（フィルタ前）** | `input/madori_trace_1_preview.png` | 部屋名・設備名・文字ラベルが含まれる原図。フィルタ後SVGでは文字が消えて空間的文脈が失われるため、フィルタ前の原図で補完する |

#### 2. メタデータ（テキスト）

| データ | ファイル | 内容 |
|---|---|---|
| **パスBBox情報** | `output/{stem}_paths.json` | `floor-plan-filter` Step 1で生成済み。各パスのID・BBox（x, y, width, height, cx, cy） |

#### 3. 分類プロンプト（指示）

- 分類カテゴリの定義（壁・ドア・窓・設備・柱・その他）
- 各カテゴリの視覚的特徴（例: ドア = 扇形の弧線、設備 = 矩形・楕円の器具形状）
- 出力形式: `{ "壁": ["path_001", ...], "ドア": ["path_045", ...], ... }` のJSON

### スキル化

分類処理は入出力が明確で自己完結するタスクのため、PoC段階からスキル（`floor-plan-classify`）として実装し、精度を検証する。

- 入出力が明確（画像+JSON → 分類JSON）で再利用可能
- `floor-plan-filter`（フィルタリング）と対になるスキルとして自然
- 分類ミス時にスキルだけ再実行できる

スキルの入出力仕様：

```
入力:
  - annotated_png: アノテーション付きSVG画像パス
  - original_png: フィルタ前の間取り原図パス
  - paths_json: パスBBox情報JSONパス
出力:
  - classification.json: { "壁": [...], "ドア": [...], "設備": [...], ... }
```

### 手順

1. `floor-plan-filter` Step 1（パスID付与 + アノテーション画像生成）— **既存データで実施済み**
2. `floor-plan-classify` スキルを実行: アノテーション画像 + 元の間取り原図 + paths.json → classification.json
3. `svg_path_analyzer.py keep` でカテゴリ別SVGを生成（例: `_walls.svg`, `_doors.svg`, `_fixtures.svg`）
4. カテゴリ別SVGをBlenderにインポートし、対応するコレクションに配置
5. カテゴリ別にextrude・マテリアルを適用して `.blend` + `.glb` を出力

### Step 3〜5: Blender側の処理詳細

#### カテゴリ別のextrude設定

| カテゴリ | extrude高さ | 根拠 |
|---|---|---|
| **壁** | 2.5m | 日本のマンション標準天井高（2.4〜2.5m） |
| **ドア** | 2.0m | 一般的なドア高さ（1.8〜2.0m） |
| **窓** | 未実装 | potraceのSVGに窓の表現がないため、PoCでは対象外 |
| **設備** | 0.8m | 浴槽・トイレ・キッチン台の概算高さ。正確な高さは設備ごとに異なるが、PoCでは一律とする |
| **柱** | 2.5m | 壁と同じ高さ |
| **その他（文字・記号等）** | インポートしない | 3Dモデルに不要 |

#### マテリアル設定

自動で一律適用する。PoCでは `diffuse_color` のみ（ノードベースのマテリアルは不要）。

| カテゴリ | 色 | 用途 |
|---|---|---|
| 壁 | オフホワイト `(0.85, 0.85, 0.82)` | 目視確認時に壁として認識しやすい |
| ドア | グリーン `(0.3, 0.6, 0.3)` | 壁と明確に区別 |
| 設備 | ブルー `(0.2, 0.5, 0.8)` | 設備位置の視認 |
| 柱 | レンガ色 `(0.78, 0.33, 0.16)` | 構造体として区別 |
| 床 | フローリング色 `(0.75, 0.68, 0.55)` | 壁BBoxから自動生成 |

#### 出力ファイル

| ファイル | 形式 | 用途 |
|---|---|---|
| `{stem}.blend` | Blender native | Blender GUIでの確認・編集。カメラ・ライト含む |
| `{stem}.glb` | glTF Binary | 後続パイプライン（floor_plan_to_video_sub_render等）やWebビューワーでの利用。`bpy.ops.export_scene.gltf()` で出力 |

### 期待される効果

- BBoxでは判別不可能な**ドア弧 vs 設備**をClaudeが形状から判別
- フィルタ前の原図に部屋名（「トイレ」「浴室」等）があるため、空間的文脈を理解した分類が可能
- 薄い壁セグメントも視覚的に壁と認識可能
- Blender側の処理がシンプルになる（分離・分類ロジック不要、SVGインポートだけ）

### 検証計画

#### ground truth（正解データ）の作成

`madori_trace_1` に対して手作業で正解分類を作成する：

- `output/madori_trace_1_indexed_annotated.png` と `input/madori_trace_1_preview.png` を目視で照合
- 全パス（197個）に対してカテゴリを手動で割り当て
- `output/ground_truth.json` として保存（classification.jsonと同じ形式）

#### 精度評価

classification.json と ground_truth.json を突合し、カテゴリごとに precision / recall を算出する。

```
precision = 正しく分類されたパス数 / そのカテゴリに分類されたパス数
recall    = 正しく分類されたパス数 / 正解でそのカテゴリに属するパス数
```

#### 合格基準

| カテゴリ | recall 基準 | 理由 |
|---|---|---|
| **壁** | **100%**（分類漏れ = 0） | 壁が欠けると3Dモデルの構造が崩れるため、漏れは許容しない |
| ドア | ≥ 80% | 誤分類があっても3Dモデルへの影響は限定的 |
| 設備 | ≥ 80% | 同上 |
| その他 | 基準なし | 残余カテゴリ |

壁のrecall 100%を最優先とし、未達の場合は分類プロンプトを調整して再実行する。

### 未解決の課題

- パス数が多い場合（197パス）、Claudeの分類精度
