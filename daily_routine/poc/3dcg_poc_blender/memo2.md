# PoC: PNG背景付きSVGに壁rectを配置するアプローチ

## 概要

potraceの「面の塗りつぶし」SVG出力では壁の個別分離が構造的に困難だったため、potraceを使わず**PNG間取り画像をSVGに背景埋め込みし、Claudeが壁の位置に`<rect>`要素を配置する**方式を検証した。

## 背景: potraceアプローチの限界

memo.mdで検証した問題:

1. **potraceの出力構造**: 黒い領域の輪郭をeven-odd fill ruleで描画。壁・ドア弧・設備が1つの複合パス（path_001）に結合
2. **サブパス分解の試行**: path_001を37サブパスに分解（`decompose`コマンド）したが、主構造（sub_001, 219セグメント）が外壁+内壁の巨大な1パスのまま残る
3. **分解後の描画崩壊**: even-odd fill ruleの関係が崩れ、部屋空間が黒く塗りつぶされる
4. **セグメントレベルの分離も困難**: 直線(`l`)と曲線(`c`)で壁/ドアを区別しようとしたが、設備にも直線が含まれるため判別不可能

### 試行した画像処理アプローチ（memo.mdより）

| アプローチ | 結果 | 問題 |
|-----------|------|------|
| fill → Loose Parts | 39オブジェクト | 主構造が1〜3個の大オブジェクトに留まる |
| スケルトン化 → potrace | 3パス | 壁が交差点で繋がったまま |
| エッジ検出 → potrace | 38パス | 主構造が1〜2パスに集約 |
| Claude画像認識で分類 | パスレベルでは分離不可 | path_001に壁+ドア+設備が混在 |
| サブパス分解 | 37サブパス | sub_001が巨大なまま、描画崩壊 |

**結論**: potraceの「塗りつぶし面」出力から壁を自動分離するのは構造的に不可能。入力側を変える必要がある。

## 新アプローチ: PNG背景付きSVGテンプレート

### パイプライン

```
PNG間取り図
  ↓ svg_path_analyzer.py template（背景埋め込みSVGテンプレート生成）
  ↓ Claudeが画像を見て壁rectを配置
  ↓ 人間がInkscape等で目視確認・微調整（背景画像をガイドに）
  ↓ Blenderインポート（image要素は無視、rect要素のみ取り込まれる）
  ↓ extrude → 3Dモデル
```

### 実装内容

`scripts/svg_path_analyzer.py`に`template`サブコマンドを追加:

```bash
uv run scripts/svg_path_analyzer.py template input.png -o output_dir
```

- PNG画像をbase64エンコードしてSVGに埋め込み（opacity=0.3）
- viewBoxをPNGピクセルサイズに合わせる（ピクセル座標=SVG座標）
- カテゴリ別の空`<g>`グループ（walls, doors, fixtures）を事前配置

### 出力SVGの構造

```xml
<svg viewBox="0 0 856 1740" width="856" height="1740">
  <image href="data:image/png;base64,..." opacity="0.3"/>
  <g id="walls">
    <rect id="wall_001" x="144" y="83" width="652" height="17"
          fill="rgba(255,0,0,0.3)" stroke="red" data-label="北側外壁"/>
    ...
  </g>
  <g id="doors"/>
  <g id="fixtures"/>
</svg>
```

### Blenderとの互換性

| SVG要素 | Blenderインポート |
|---------|-----------------|
| `<rect>` | **対応** — 4点の閉じたパスに変換 |
| `<image>` | **無視** — 背景画像はスキップされる |

→ 背景画像は視覚的確認用のみ。Blenderには壁rectだけが取り込まれる。

## 検証結果

### 試行1: 元の間取り原図（madori_trace_1_preview.png, 548×1113px）を背景に使用

- 27個の壁rectを配置
- 外壁は概ね合っているが、テキストや設備が重なって壁の輪郭が見えにくい
- 精度: 中程度

### 試行2: フィルタ済み壁線画像（madori_trace_1_indexed_walls_preview.png, 856×1740px）を背景に使用

- 37個の壁rectを配置
- テキストが除去されて壁線がくっきり見えるため、壁位置の特定精度が大幅向上
- ピクセルスキャンデータ（水平・垂直方向の黒ピクセル塊検出）を併用して座標を精密化
- 精度: 高（外壁はピクセルレベルで合致、内壁の主要仕切りもカバー）

### 背景画像の選択

| 背景画像 | 解像度 | メリット | デメリット |
|---------|--------|---------|-----------|
| 原図PNG | 548×1113 | 部屋名・設備名が見える | 壁とテキストが重なり見にくい |
| フィルタ済みPNG | 856×1740 | 壁線のみでくっきり | 部屋名が消えている |

**推奨**: フィルタ済みPNGの方が壁rect配置の精度が高い。部屋名が必要な場合は原図を別途参照する。

## メリット

1. **個別分離が最初から実現**: 各壁が独立した`<rect>`要素（potraceの複合パス問題が発生しない）
2. **修正が容易**: Inkscapeで背景画像を見ながらrectをドラッグするだけ
3. **壁以外の要素を含まない**: ドア弧・設備は別グループ（walls/doors/fixtures）で管理
4. **Blender互換性**: `<rect>`はBlenderのSVGインポーターで直接読み込み可能

## 課題・制約

1. **座標精度**: Claudeの画像認識によるrect配置は完璧ではない。人間の微調整が前提
2. **ピクセルスキャン補助**: ピクセルデータ分析で精度を補えるが、完全自動ではない
3. **背景画像の前処理が必要**: フィルタ済み画像（floor-plan-filter出力）が必要
4. **スケーリング**: SVG座標はピクセル単位。Blenderでの実寸変換は別途必要

## 改善方針

今回のPoCでは「Claudeが画像を見て壁rectを配置」したが、以下の改善が必要:

### 1. 最初にピクセル値を読みに行く

Claudeの画像認識で座標を推測するのではなく、**ピクセルデータの分析を先に行う**。今回の試行でもピクセルスキャン（水平・垂直方向の黒ピクセル塊検出）を併用したところ精度が大幅に向上した。画像認識による座標推定は精度が低いため、ピクセル値の読み取りをファーストステップとする。

### 2. 特徴点ベースのアプローチ

壁の配置を「長方形をどこに置くか」ではなく、**2段階のタスクに分解**する:

```
Step 1: 特徴点の検出（ピクセルデータから）
  - 壁の端点（壁が途切れる場所）
  - 壁の交差点（T字・L字・十字の接合部）
  - 壁の角（外壁のコーナー）

Step 2: 特徴点同士を繋ぐ
  - 2つの特徴点を結ぶ線 = 1つの壁セグメント
  - 線に壁厚を付与 → 長方形rect
```

これにより:
- 特徴点の検出はピクセルデータから機械的に可能（交差点=周囲に3方向以上の黒ピクセル）
- 「どの特徴点同士を繋ぐか」の判断にClaudeの画像認識を使う（人間の判断に近いタスク）
- 座標精度はピクセルデータが担保し、構造判断はClaude/人間が担当する役割分担

### 3. 反復ループ処理としての設計

壁rect配置は1回で完了するタスクではなく、**繰り返しの修正ループ**が必要:

```
特徴点検出 → rect配置 → プレビュー確認 → 修正指示 → rect調整 → プレビュー確認 → ...
```

このループを毎回手動で指示するのはコストが高いため、**スキル化**して一連のフローを定型化する。skill-creatorで作成する。
