# 調査メモ: 3DCG/VFXツール調査（Houdini vs Blender）

## 調査のきっかけ

Instagramで見つけた3DCG/VFX投稿（research/1/*.png）を分析。骸骨キャラクターが山岳環境でミサイル破壊・シールド展開するシネマティックなVFX作品。

## 投稿の技術分析

全てHoudini FXによる物理シミュレーションで制作されている（生成AIは不使用）。

| 要素 | 技術 | Houdiniモジュール |
|---|---|---|
| ミサイル破壊 | Custom cluster and fracture | RBD（剛体破壊シム） |
| 爆発・衝撃波 | Pyro shockwave interaction with trees | Pyro（ボリュメトリック流体シム） |
| 樹木の変形 | Vellum Trees | Vellum（ソフトボディソルバー） |
| シールドエフェクト | CHOPs Shield | CHOPs（チャンネルオペレータ） |
| 破片・パーティクル | Shield Debris / Shield Particle | POP（パーティクルシム） |
| 筋肉変形 | Arm Muscle Expansion | Muscle系 / カスタムデフォーマ |

## Houdini FXとは

SideFX社の3DCG/VFXソフトウェア。物理シミュレーション・プロシージャル生成に特化。映画VFX業界のデファクトスタンダード。

最大の特徴は**プロシージャルワークフロー**:
- 全操作がノードグラフで構築される
- パラメータを変えれば結果が自動更新（非破壊）
- 「手で作る」ではなく「作り方のレシピを組む」

## Blender vs Houdini

### 守備範囲の違い

- **Blender** = 3DCGの万能ツール。モデリング・アニメーション・レンダリング等が全部入り
- **Houdini** = 物理シミュレーション・プロシージャル生成の専門ツール

```
工程                Blender    Houdini
───────────────────────────────────────
モデリング            ◎          △
スカルプト            ◎          ✕
テクスチャ/UV         ○          △
リギング              ○          ○
アニメーション         ○          △
物理シミュレーション    △〜○       ◎◎
プロシージャル生成      △〜○       ◎◎
レンダリング           ◎          ○
動画編集              ○          ✕
```

### 操作の根本的な違い

- **Blender**: 直接操作型。マウスで頂点を動かす。Photoshop的
- **Houdini**: プロシージャル型。ノードで処理手順を定義。コード的

## 学習ロードマップ

```
Step 1: Blenderで「世界を作る」力をつける
  モデリング、マテリアル、ライティング、カメラ、レンダリング
  → 既存bpyパイプラインを拡張

Step 2: HoudiniでVFXを加える
  物理シミュレーション（破壊・流体・煙・パーティクル）
  → Blenderで作った世界に動きと物理的リアリティを加える
```

## Blender ↔ Houdini 連携方法

ファイルベース連携が基本。

| 形式 | 用途 | 特徴 |
|---|---|---|
| Alembic (.abc) | メッシュ・アニメーション | 業界標準。両方ネイティブ対応 |
| USD (.usd) | シーン全体 | 次世代の標準 |
| OpenVDB (.vdb) | ボリューム（煙・炎） | Houdini→Blenderの煙受け渡し |

### Claude Codeパイプラインでの構成

```
Claude Code（オーケストレーター）
  │
  ├─ run_blender.sh → bpyスクリプト
  │    モデリング・シーン構築 → Alembic書き出し
  │
  ├─ hython → houスクリプト
  │    Alembic読み込み → 物理シミュレーション → Alembic/VDB書き出し
  │
  └─ run_blender.sh → bpyスクリプト
       シミュレーション結果読み込み → 最終レンダリング
```

## ビジョンとの接点

Houdiniの「ノードで処理手順を定義→パラメータで制御→バッチ実行」という設計思想は、Claude Codeでパイプラインを組む思想と一致する。Blenderは手動ツールをスクリプトで無理やりパイプライン化しているのに対し、Houdiniは最初からパイプラインツールとして設計されている。

Houdiniは職人芸だが、Claude Code等のAIエージェントを使ってワークフロー化・パイプライン化することで、職人でなくてもエンジニアリングで解決できる可能性がある。
