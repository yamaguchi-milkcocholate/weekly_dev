# Blender Level Up ノート

## 目標

Claude Codeを通じてBlenderを対話的に操作し、頭の中の世界や生成AIで拡張されたアイデアを3DCGとして実現する。
作りたいものを題材にして学び、繰り返す作業をClaude Codeで自動化していく。

## 根本的な気づき（2026-04-09）

### AIと3DCGスキルの関係

プログラミングの世界で起きていることと同じ構造が3DCGにもある:
- **素人 + AI = 変なものしか作れない**（今日の実験で証明済み）
- **スキルがある人 + AI = 10倍の生産性**
- AIが不要にするのは「新規参入者の低い壁」であり、スキルを持つ人間はより拡張される

### Claude Codeの正しい位置づけ

- Claude Codeは3DCGの**羅針盤を持っていない**。Web検索で集めた情報を並べるだけでは正確さ30-40%程度
- 「こうすればできる」と技術を羅列しても、それが本当に正しいか検証していない
- **Claude Codeの役割**: Blenderがわかる人が作業を自動化・拡張するためのツール

### Claude Codeが羅針盤を持つための方法

Claude Code単体では3DCGの判断力を持てない。しかし:
- **ユーザーが教材（書籍・プロのワークフロー解説等）を渡せば、そこから適切な知識を抽出できる**
- プログラミングでいう「公式ドキュメントやオライリー本を渡して、必要な情報を引き出す」と同じ構造
- 教材の質 = Claude Codeの判断の質

### 教材の準備方針

Claude Codeが処理できる形式:
- **直接読める**: Blender公式マニュアル、書籍（PDF）、.blendファイル（bpyで解析）、テキスト記事
- **直接読めない**: 動画チュートリアル（3DCG教育の主流はほぼ動画）
  - 対策: ユーザーが動画を見て学ぶ → 要点やつまずきをClaude Codeに共有 → 構造化・記録

**次のアクション**: Blender環境アート系の体系的な教材（書籍・コース）を選定し、Claude Codeに渡す。

## 方針

- **作りたいものを先に決め、それを実現する過程で学ぶ**（教科書的な段階学習はしない）
- Claude Codeを通じてBlenderを対話的に操作する環境を構築する
- アドオンやツールで効率化できる方法をClaude Codeが調査・提案する
- 自分が学ぶ過程で、Claude Codeで自動化・拡張できるポイントを見つけていく

### 3DCG学習に投資する理由（2026-04-13 議論で確定）

Blender側の情報量がAI生成の品質を決める。

```
Blender側の情報量が少ない → AIの自由度が高い → 思い通りにならない
Blender側の情報量が多い → AIの自由度が低い → 意図通りになる
```

間取りパイプライン（PoC 0-9）で実証済み:
- 壁・柱のみ + Tripoアセット（テクスチャなし・小物なし）でV2Vに渡した結果、AIが自由に解釈し品質が不安定になった
- テクスチャ・小物・ライティングをBlender側で作り込むほど、AIの解釈の余地が減り、意図通りの結果が得られる

**3DCGスキルを上げる = Blender側の情報量を増やせる = AI生成の品質が安定する。**

これは創作パイプラインでも同じ。キャラクターのポーズ・衣装・背景をどこまで詳細に作るかが最終出力の品質を決める。

### ツール分担

| ツール | 役割 |
|--------|------|
| **Gemini（WebUI）** | YouTube動画の調査・分析。Blenderチュートリアルから手順・アドオン・テクニックを抽出 |
| **Claude Code** | 抽出された情報をbpyスクリプト化、対話的Blender操作、反復作業の自動化 |
| **Blender（GUI）** | 人間が直接操作して学ぶ。Sculptやカメラ構図など判断が必要な作業 |

### Claude Codeに求める役割

1. **対話的操作**: 自然言語の指示 → bpyスクリプト生成・実行 → 結果を見てフィードバック
2. **効率化手法の調査**: 公式ドキュメント・データソースから最適な方法を提案
3. **検証済み知識からの抽出**: 教材・公式ドキュメントを渡し、必要な情報を引き出す
4. **反復作業の自動化**: 繰り返す作業パターンをスキル化する

### 学習情報の流れ

```
Gemini（YouTube動画分析）→ 手順・テクニック抽出
    ↓ 人間がClaude Codeに共有
Claude Code → bpyスクリプト化 / 手順整理 / 自動化
    ↓
Blender → 実行・確認
```

## Step 1: 地形（山塊と湾） + 海 + 空

### 目的

- 「山が海に落ち込む湾」の全体シルエットをプロシージャルに作る
- reference画像の構図（山の中腹から見下ろす湾の全景）を再現する
- seed値やパラメータを振って複数バリエーションを一括生成する

### 学ぶ技術

1. **Geometry Nodes**: ノードツリーをbpyで構築し、Set Position + Noise Textureで地形生成
2. **ノイズレイヤリング**: 大地形（Perlin）→ 中間（Voronoi）→ 細部の重ね合わせ
3. **Ocean Modifier**: 海面を地形の下に配置（交差線 = 海岸線）
4. **Nishita Sky**: Blender 5.0では `SINGLE_SCATTERING` （旧NISHITA）
5. **Cycles + OIDN**: フォトリアルレンダリング + デノイジング

### Blender 5.0 API メモ

- Sky Texture: `sky_type = "SINGLE_SCATTERING"`（旧NISHITA）
- Sky Texture: `aerosol_density`（旧dust_density）
- Ocean Modifier: `wind_velocity`（旧wind_speed）
- `World.use_nodes` は Blender 6.0 で削除予定（DeprecationWarning）

### 作業ログ

#### 試行1: 空と海のみ（失敗）

- `step1/generate_sky_ocean.py` で太陽角度3パターン x 波スケール3パターン = 9枚生成
- 問題: 地形がないため構図が成立しない。水面ギリギリの視点になり、reference画像の「山の中腹から見下ろす湾の全景」と全く異なる
- 学び: **この風景は地形がないと成立しない。空と海を切り出して先に作るのは間違い**

#### 試行2: Geometry Nodes + bpyスクリプトでプロシージャル地形生成

- Geometry Nodesでノイズレイヤリング + 関数的な地形（Y方向傾斜 × X方向尾根パターン）で山塊 + 湾の地形を生成
- Ocean Modifier付きPlaneを海面（Z=0）に配置 → 地形との交差線が海岸線
- カメラは山の中腹から見下ろす位置に配置
- `.blend`ファイルを保存してGUIで確認可能にした
- 結果: 山・湾・海の基本構造は再現できた（`step1/output/exp1/ita_terrain.blend`）

##### つまずいたポイント
1. **Noise Textureのスケール問題**: ワールド座標（数百m）に対してScale値が大きすぎるとスパイク状、小さすぎると平坦になる。座標を正規化してからノイズに渡す必要がある
2. **Subdivide Meshの過剰分割**: Level 8だと頂点数が爆発。代わりに`primitive_grid_add(x_subdivisions=256)`で最初から高解像度メッシュを作る方が安定
3. **Principled BSDFの取得**: Blender 5.0では`nodes.get("Principled BSDF")`が`None`を返す場合がある。`nodes.clear()`して手動で作り直すのが確実
4. **高さスケール**: TERRAIN_HEIGHT=80では起伏が小さすぎた。200に上げて初めて山らしくなった

#### 方針転換: Blenderの基礎スキル習得を優先する

##### 気づき

- **Claude Codeだけでは地形を作れない**: 関数的な地形設計やパラメータ調整には、Blenderの基本的な知識（Geometry Nodes、モディファイア、マテリアル、カメラの挙動）が前提として必要
- **Claude Codeの役割**: ある程度Blenderが使える人が、作業を自動化・拡張するためのツール。全くの素人がClaude Codeだけで高品質な3DCGを作るのは現実的ではない
- **正しい学習順序**: まずBlenderを自分で使えるようになる → その中でClaude Codeで拡張できる部分を探す

##### 次のアクション

Blenderで地形を作る場合の**手動での作業手順**を調査・整理する。GUIベースのワークフローを理解した上で、Claude Codeで自動化できるポイントを見極める。

### Blenderで自然地形を作る作業手順

#### 3つのアプローチ

| アプローチ | 難易度 | 特徴 |
|-----------|--------|------|
| A.N.T. Landscape | 初心者 | Extensions有効化 → Add > Mesh > Landscape。数分で地形完成。非破壊でない |
| Displace Modifier | 初心者 | Plane → Subdivision Surface → Displace Modifier + テクスチャ。シンプルだが侵食表現が困難 |
| **Geometry Nodes** | **中級者** | **完全非破壊、パラメータ変更で即更新。bpy自動化と最も相性が良い** |

#### プロ推奨ワークフロー（ハイブリッド）

1. **A.N.T. Landscape or Geometry Nodes** でベース地形を生成
2. **Sculpt Mode** で意図的な形状調整（湾の形、山の位置など）
3. **Geometry Nodes** で侵食・ディテール・マテリアル分岐を追加
4. **Ocean Modifier** で海面
5. **Geometry Nodes Scatter** で植生配置
6. 参考写真を Background Image に設定しシルエットを合わせる

**ポイント**: ステップ2の**Sculpt Mode**がキー。プロシージャルで大まかな形を作り、手動で意図した形に調整する。これが「井田の地形を再現する」のに必要なステップ。

#### Claude Codeで自動化できる部分 / 手動の方が良い部分

| ステップ | Claude Code | 理由 |
|---------|-------------|------|
| ベースメッシュ作成 | **自動化向き** | パラメータ指定のみ |
| ノイズレイヤリング（Geometry Nodes構築） | **自動化向き** | ノードツリー全体をbpyで構築可能 |
| A.N.T. Landscape呼び出し | **自動化向き** | `bpy.ops.mesh.landscape()` 一発 |
| seedバリエーション生成 | **自動化向き** | ループでseed変えてレンダリング |
| 海面(Ocean Modifier) | **自動化向き** | Modifier追加+パラメータ設定 |
| マテリアル・ライティング | **自動化向き** | Shader Nodes構築がbpyで可能 |
| **Sculptによる手動造形** | **手動向き** | ブラシストロークの自動化は非現実的 |
| **シルエット・構図の評価** | **手動向き** | アーティスティック判断が必要 |

#### 学習の進め方

1. まずBlender GUIで基本操作を覚える（ナビゲーション、オブジェクト操作、モディファイア）
2. Geometry Nodesの基本を理解する（GUIで）
3. その上で、Claude Codeで自動化できるポイントを特定し、スキル化する

## 方針転換: 対話的Blender操作（2026-04-13）

### 背景

v2ロードマップ（Houdini + Blenderのマルチツールパイプライン）を策定したが、議論の結果、以下が判明:

1. **創作の目的はSNS発信**（アニメシーン再現・創作、物語の映像化） → Houdiniの地形・侵食シミュレーションは不要
2. **AI生成に任せすぎると品質が不安定**（間取りパイプラインの経験） → 3DCG側の制御力が必要
3. **参考クリエイターはHoudini/Unrealも使っている** → シミュレーション（流体・破壊等）が必要になった段階で導入を検討
4. **まずBlenderの対話的操作を確立する**のが最優先

### 方針

Claude Codeを通じてBlenderを対話的に操作する環境を構築する。

```
人間: 意図を言語化する（What）
Claude Code: bpyスクリプト生成・実行（How）
人間: 結果を評価・選択する（Judge）
```

### 学ぶべきスキル

| スキル | 用途 |
|--------|------|
| シーン構築（環境・小物） | 舞台を作る |
| カメラワーク | 映画的な演出 |
| ライティング | 雰囲気・時間帯の表現 |
| マテリアル | 質感の表現 |
| キャラクター表現 | ポーズ・シルエット |

### v2ロードマップを廃止した理由

- Houdini学習は事業戦略上の創作目的に対してオーバースペック
- 地形生成・侵食シミュレーションはSNS創作のユースケースに合わない
- ツール調査（`../docs/memo/terrain_tool_research.md`）は参考資料として保持するが、学習ロードマップとしては不採用
