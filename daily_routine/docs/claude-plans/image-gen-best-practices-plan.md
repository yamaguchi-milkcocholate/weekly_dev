# 画像生成ベストプラクティス検証計画

## 背景

PoC Phase 1〜4 で Kontext / Gemini / Runway の基礎的な比較検証を実施した。Gemini 3 Pro が全タスクで高品質・最安という結論を得たが、各タスクのプロンプトや手法は未最適化のまま。

本計画では、Gemini を Core としつつ、タスクごとにベストプラクティスを確立する。

## ゴール

`docs/image_gen_best_practices/` に、タスクごとの検証済みベストプラクティスを蓄積する。

各ベストプラクティスは以下を含む:

- 推奨 API・方式
- 最適化されたプロンプトテンプレート
- 注意点・アンチパターン
- 検証エビデンスへのリンク

## 利用シーンの再確認

```
インフルエンサーのvlogキャプチャ（seed画像）
  ↓ 画角・雰囲気・構成を参考にしつつ
独自キャラクター + 独自シナリオ + 独自環境 で再構成
  ↓
キーフレーム画像群 → 動画化
```

典型的な1カットの生成フロー:

```
seed画像 → [人物差し替え + ポーズ調整 + 背景変更] → キーフレーム画像
```

## 検証フェーズ設計

### Phase A: 単体タスク最適化（B1, B2, B3, B5, B6）

各タスクを個別に最適化する。フェーズ内は独立しているため並行可能。

#### Phase A-1: 人物差し替え（B1）精度向上 — **完了**

**目的**: Gemini での人物差し替えプロンプトを最適化し、ベストプラクティスを確立する

**結果**: 2回の検証を実施し、ベストプラクティスを確立した。

**初回検証（テキスト指示のみ、REST API）**:

- 8パターン × 3 seed = 24枚生成、$0.96
- テキスト指示のみでは P3（選択的編集テンプレート）と P6（ALL CAPS）が最高評価
- ただし参照画像なしではキーフレーム間のキャラクター一貫性が保てない課題を発見
- P5（参照画像あり）はキャラ一致度は良好だったが、REST API の制限でアスペクト比が崩壊

**再検証（参照画像ベース、Python SDK）**:

- 6パターン × 3 seed = 18枚生成、$0.72
- Python SDK の `ImageConfig(aspect_ratio="9:16")` でアスペクト比問題を解決
- 全パターンで参照画像を使用し、プロンプト要素の組み合わせを検証
- **結論: 全部入り（R6）が最強** — 参照画像 + Identity Block + ALL CAPS + 画像役割記述

**実コスト**: $1.68（初回 $0.96 + 再検証 $0.72）

**成果物**: `docs/image_gen_best_practices/character_swap.md`

**横断的知見（他フェーズへの影響）**:

- REST API ではなく Python SDK（`google-genai`）を使用すること
- 参照画像は「逆効果」ではなく「SDK で正しく使えば有効」
- ALL CAPS 保持指示（`MUST` / `MUST NOT`）は背景保持に効果的

#### Phase A-2: ポーズ変更（B2）— 2 段階 AI ワークフロー — **完了**

**目的**: 任意のポーズ変更指示に対して、何を変えて何を変えないかを適切に判断し、プロンプトに反映する設計思想・フローを確立する

**v1: 初回検証（ポーズ記述粒度の比較）**:

- 7パターン × 3 seed = 21枚生成、$0.84
- ポーズ記述の粒度（簡潔 vs 部位別詳細 vs 段階的記述）を比較
- **知見**: ポーズの伝え方よりも「画角が変わる」「座り位置が変わる」等の意図しない構図変化を防ぐ制約の設計が本質的な課題だと判明
- 生成サンプル: `poc/seamless/generated/phase_a2/`

**v2: 再検証（制約設計）**:

- 8パターン × 3 seed = 24枚生成、$0.96
- 制約の有無・種類を変えて検証（制約なし / 位置固定 / 画角固定 / フル制約 / モデル委任型 等）
- 2種のポーズ（自撮り・腕組み）と意図的な位置変更（立ち上がり）も検証

**v2 の主要な発見**:

1. **P4/P6（フル制約）が最高品質**: seed 固有の具体的な制約（`While remaining seated`、`objects on the desk`）で構図・オブジェクトが高精度に保持される
2. **P8（モデル委任型）は品質不足**: 致命的失敗はないが、机の形状変化など細部の保持が不十分。及第点に達しない
3. **P2（位置ハードコード）に致命的欠陥**: `While remaining in her current seated position` は立ち seed で座りに強制変更
4. **P3（画角固定のみ）は不十分**: 画角指示だけでは位置変更（座り→立ち）を防げない
5. **制約はポーズ種類に依存しない**: 腕組み（P6）でも自撮り（P4）と同等にフル制約が機能

**v2 の結論**:

- フル制約（P4/P6）が品質基準を満たす唯一のパターン
- ただしフル制約には seed 固有の記述が必要なため、パイプラインでは **2 段階 AI ワークフロー** で自動化する:
  - **Step 1（分析）**: Gemini が seed 画像を分析 → seed 固有の制約付きプロンプトを生成
  - **Step 2（加工）**: Gemini がそのプロンプトで seed 画像を加工
- ただしフル制約には seed 固有の記述が必要なため、パイプラインでは **2 段階 AI ワークフロー** で自動化する:
  - **Step 1（分析）**: Gemini Flash が seed 画像を分析 → シーン記述型プロンプトを生成
  - **Step 2（加工）**: Gemini Pro がそのプロンプトで seed 画像を加工

**v3: Step 1 メタプロンプトの検証**:

- 4 パターン × 3 seed = 12 枚生成、$0.60
- Step 1: `gemini-3-flash-preview`、Step 2: `gemini-3-pro-image-preview`
- メタプロンプトの指示方法を比較（最小指示 / 分析項目明示 / テンプレート提示 / Few-shot）

**v3 の結論**:

- **M1（最小指示）が最良**: 分析を Gemini に全て委ね、シーン記述型のプロンプトを生成させる
- M2〜M4 は制約を明示的に列挙するため、不要な保持指示の混入・意図しない制約をモデルにかけるリスクがある
- 全パターンで画像品質は A 以上。2 段階ワークフロー自体の有効性が実証された
- 位置判定（座り/立ち）は全パターンで 100% 正確

**実コスト**: $2.40（v1 $0.84 + v2 $0.96 + v3 $0.60）

**成果物**: `docs/image_gen_best_practices/pose_change.md`

**横断的知見（他フェーズへの影響）**:

- **ハードコード制約は危険**: seed の状況が多様な場合、固定の位置指示は逆効果
- **2 段階 AI ワークフローは汎用的**: B3（背景変更）や B4（複合編集）にも同様の分析→加工フローを適用可能
- **メタプロンプトは最小指示が最良**: 制約を列挙すると意図しない制約をかけるリスクがある。モデルに自由に分析させる方が安全
- ポーズ指示は動作名を 1 文で簡潔に書く（部位別詳細・段階的記述は逆効果）
- カメラアングルのレンズ・ショットタイプ指示は I2I 編集では構図変化を引き起こす

#### Phase A-3: 背景変更（B3）精度向上 — **完了**

**目的**: 人物を完全に維持しつつ背景のみを高品質に差し替えるプロンプトを最適化する

**結果**: 8パターン × 3 seed = 24枚生成（$0.96）。空間認識を最重要基準として評価。

**検証設計**: 全パターンでキャラ参照 + seed + 背景参照（桜並木）の3枚入力。テキスト指示の組み合わせを変えて検証。

**結論: B2（画像役割記述）が最強** — 3枚の画像役割を明示 + 簡潔な背景置換指示 + 人物保持指示。空間認識・人物保持・テキストオーバーレイ保持の全てで最も安定。

**最重要発見 — 編集モード vs 生成モード**:
- `Replace the background`（編集モード）: 人物がその環境に実在するように自然に溶け込む
- Identity Block / `Place into`（生成モード）: 人物と背景が別々に生成され、バーチャル背景のような壁紙感になる
- **人物差し替え（A-1）では「全部入り最強」だったが、背景変更では Identity Block が逆効果**。タスクによって最適手法が正反対

**実コスト**: $0.96

**成果物**: `docs/image_gen_best_practices/background_change.md`

**横断的知見（他フェーズへの影響）**:
- タスクの性質（何を変えて何を保持するか）によって、Identity Block の要否が逆転する
- モデルを「編集モード」で動作させるには、`Replace X` 系の動詞を使い、人物をテキストで記述しないこと
- 3枚以上の画像を入力する場合は、各画像の役割を冒頭で明示すると安定性が上がる

#### Phase A-4: スタイル転写（B5）— **不要と判断、スキップ**

**目的**: seed の画角・色味・雰囲気を、独自キャラ・独自環境の新規画像に反映する手法を確立する

**不要と判断した理由**: A-1（人物差し替え）+ A-2（ポーズ変更）+ A-3（背景変更）の I2I パイプラインで全シーンをカバーできる。seed にない環境が必要な場合も A-3 で背景を差し替えれば済み、seed のトーンは I2I で自然に保持される。T2I でスタイルを転写する必要がない。

**実コスト**: $0.96（検証実施後に不要と判断）
**生成サンプル**: `poc/seamless/generated/phase_a4/`

#### Phase A-5: テキスト/オーバーレイ除去（B6）

**目的**: seed 画像上の時刻表示・テロップ等を自然に除去する手法を確立する

**ステップ**:

1. Web 調査: Gemini での inpainting / テキスト除去、他ツール（LaMa 等）との比較
2. 実験パターン:
   - Gemini I2I でテキスト除去指示
   - 前処理として別ツール（LaMa / OpenAI 等）で除去してから Gemini に渡す
   - テキストを無視して生成（除去せず上書き的に処理）
3. 評価: 除去の完全性、周辺の自然さ
4. ベストプラクティス記載

**コスト見積もり**: ~$0.48

#### Phase B-1: 複合編集（B4）— 1パス vs 分割

**目的**: 人物 + ポーズ + 背景を同時に変更する際の最適な方式を確立する

**前提**: Phase 1-2 では「1パスが最良」だったが、Phase A で各タスクの最適プロンプトが確立された後の再検証

**ステップ**:

1. 1パス方式: Phase A の各タスクの最適プロンプトを統合した1プロンプト
2. 2パス方式: [B6テキスト除去 →] B1人物差し替え+B2ポーズ変更（1パス目）→ B3背景変更（2パス目）
3. 3パス方式: [B6テキスト除去 →] B1人物差し替え → B2ポーズ変更 → B3背景変更
4. 変更度合い別の検証: 軽微な変更（ポーズ微調整のみ）vs 大幅な変更（全要素変更）

**入力画像**: seed キャプチャ 3〜5種（変更度合いの異なるシナリオ）

**評価**: 各要素の品質 + 総合的な自然さ + コスト効率

**コスト見積もり**: 4方式 × 5画像 × $0.04〜0.12 = ~$2.40

## Phase A-1 からの横断的知見

以降のフェーズで共通適用する知見:

- **Python SDK（`google-genai`）を使用する**: REST API では `aspect_ratio` を制御できない。参照画像使用時にアスペクト比が崩壊する
- **参照画像は有効**: 「逆効果」は REST API の制限が原因。SDK で正しく使えばキャラクター一貫性に貢献
- **ALL CAPS 保持指示が有効**: `MUST` / `MUST NOT` で変更範囲を厳密に制御できる
- **Identity Block は服装反映に必須**: 参照画像だけでは服装が seed のまま残る傾向
- **全部入りが最強**: 要素を減らすと品質が下がる。情報量が多い方が安定する

## 各フェーズの進め方

各フェーズは以下の3ステップで進める:

### Step 1: Web 調査

- Gemini 公式ドキュメント、画像生成コミュニティの事例を調査
- 既存の `poc/seamless/plan.md` §10 の調査結果も活用
- 調査結果は PoC レポートに記録

### Step 2: 検証（複数試行）

- `poc/seamless/` 配下に実験スクリプト・生成画像を配置
- 各パターン最低3回試行し、再現性を確認
- 生成画像を目視評価（評価基準はフェーズごとに定義）
- 有望なパターンはプロンプト微調整で追加試行

### Step 3: ベストプラクティス記載

- `docs/image_gen_best_practices/{タスク名}.md` に記載
- README.md のフォーマットに従う
- 検証エビデンス（レポート・生成サンプル）へのリンクを必ず記載
