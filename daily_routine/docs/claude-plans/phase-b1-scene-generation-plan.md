# Phase B-1 改訂: シーン統合生成 — 検証計画

## 経緯と問題定義

### Phase A で確立した知見

| Phase | タスク | ベストプラクティス | 核心原理 |
|-------|--------|-------------------|----------|
| A-1 | 人物差し替え | 参照画像 + Identity Block + ALL CAPS + 画像役割記述（R6: 全部入り） | Identity Block で服装反映。参照画像で顔一貫性 |
| A-2 | ポーズ変更 | 2段階AIワークフロー（Flash分析 → Pro生成）。人物差替と1パスで同時実行 | Flash がseed固有の制約を動的に生成。ハードコード制約は危険 |
| A-3 | 背景変更 | 画像役割記述（B2）。Identity Block を**含めない** | 「Replace the background」で編集モード発動。Identity Block は生成モードを誘発 |
| A-5 | テキスト除去 | 簡潔指示（`Remove all text overlays from this image.`） | 最小限の指示が最良。追加指示は成功率を下げる |

### A-1 と A-3 の根本的矛盾

- **人物差し替え**: Identity Block が**必須**（服装を反映するため）
- **背景変更**: Identity Block が**禁止**（人物が再生成されてしまう）

この2つを同一パイプラインで実現しようとすると、どちらかの品質が犠牲になる。

### Phase B-1 パイプライン検証（v1〜v3）の結果

B-1 では A1〜A5 のベストプラクティスを**段階的パイプライン**として統合し、一気通貫で動作するかを検証した。

| バージョン | パイプライン順序 | 結果 | 原因 |
|-----------|----------------|------|------|
| v1 | テキスト除去 → 人物差替+ポーズ → 背景変更 | 背景変更がバーチャル背景化 | AI生成画像を背景変更に入力 → 品質劣化 |
| v2 | テキスト除去 → 背景変更 → 人物差替+ポーズ | 背景品質がA3単体より劣化 | テキスト除去(AI処理1回)後の画像を入力 |
| v3 | 背景変更(元seed直接) → テキスト除去 → 人物差替+ポーズ | 背景変更品質が不安定 | 3画像入力でキャラ参照が人物差替を誘発 / 2画像入力でもA3単体の品質に至らず |

### 結論: レイヤー合成アプローチの限界

段階的パイプライン（各タスクを順番に実行）は本質的に**レイヤー合成**である。
背景を貼り替え → 人物を貼り替え → …と進めるため、各ステップの出力が次のステップの品質を制約する。

特に問題なのは:
- 人物と環境の**照明・接地面・奥行き・空気感**が一体にならない
- AI処理を通過するたびに画質が劣化する（実写→AI生成の遷移が不可逆）
- A-3 の背景変更品質自体が非決定的で、パイプラインの安定性が確保できない

## 新アプローチ: シーン統合生成

「背景を貼り替える」から「シーンを描き起こす」へパラダイムシフトする。

人物と環境を**同時に生成**することで、照明・接地・空気感が一体で生成される。

### 方針A: 統合1パス生成

seed の構図を参照しつつ、キャラクター + ポーズ + 環境を1回のAPI呼び出しで生成する。

**利点**: 最もシンプル。人物と環境が一体で生成されるため空間認識が自然。
**懸念**: タスクが多すぎて精度が悪化する可能性（人物差替 + ポーズ変更 + 背景変更 + 構図維持を同時に要求）。

### 方針B: Flash統合分析 → Pro生成

Flash が seed + 背景参照を分析し、**環境に統合されたシーン記述**を生成。
そのプロンプトで Pro が画像を生成する。

**利点**: Flash の分析力で構図保持精度を向上。A-2 で実証済みの2段階ワークフローの応用。
**懸念**: Flash が生成するプロンプトの品質に依存。

### 方針C: 精度維持型分割

1パスのタスク過多を回避するため、分割するが「レイヤー合成」ではない分割方法を探る。

**例**: シーン生成（環境+構図）→ キャラクター統合（A-1手法で差替）
**利点**: 各ステップのタスクが絞られる。A-1 の実証済み手法を活用できる。
**懸念**: 2ステップ目のキャラ差替で再びバーチャル背景感が出る可能性。

## 検証パターン設計

### S1: 統合1パス（3画像入力、シーン生成指示）

A-1（全部入り）のプロンプト構造をベースに、背景参照を追加する。
「Replace the background」（編集モード）ではなく「Generate a scene」（生成モード）を意図的に使う。

```
入力: [キャラ参照] + [seed] + [背景参照] + テキスト
```

```
Image 1 shows the target character.
Image 2 shows the reference composition and pose.
Image 3 shows the target background environment.
Generate a photo of the character from image 1 in the environment from image 3,
matching the composition and camera angle from image 2.
The character is: {Identity Block}
The character's pose: {pose_instruction}
MUST match the spatial layout, depth, and camera angle from image 2.
MUST place the character naturally within the environment from image 3.
Single person only, solo.
```

**検証ポイント**: 3つの参照画像を同時に処理できるか。構図保持と環境統合の両立。

### S2: 統合1パス（2画像: キャラ参照 + 背景参照、構図はテキスト記述）

seed 画像を入力せず、Flash が事前にseedを分析して構図をテキスト化。
Pro は キャラ参照 + 背景参照 + テキストのみで生成。

```
Step 0: Flash が seed を分析 → 構図記述テキスト（カメラアングル、人物の位置、前景オブジェクト等）
Step 1: Pro が [キャラ参照] + [背景参照] + 構図記述 + Identity Block + ポーズ指示 で生成
```

**検証ポイント**: seed なしでも構図を再現できるか。テキスト記述の精度。

### S3: Flash統合分析 → Pro生成（3画像入力）

A-2 の2段階ワークフローを拡張。Flash が seed + 背景参照の**両方**を分析し、統合シーンプロンプトを生成。

```
Step 1: Flash が [seed] + [背景参照] を分析 → 統合シーン記述を生成
  メタプロンプト:
    「Image 1 は元の構図。Image 2 はターゲット環境。
     Image 1 の構図・カメラアングル・前景オブジェクトを維持しつつ、
     Image 2 の環境に配置するシーン記述を生成せよ。
     キャラクター: {Identity Block}
     ポーズ: {pose_instruction}
     出力: Image 1(キャラ参照) + Image 2(シーン) の2画像入力を想定したプロンプト」

Step 2: Pro が [キャラ参照] + [背景参照] + 統合プロンプト で生成
```

**検証ポイント**: Flash が構図+環境を統合したプロンプトを生成できるか。S1 との品質比較。

### S4: Flash統合分析 → Pro生成（seed も入力に含む）

S3 と同じだが、Pro にも seed を渡して構図参照を強化する。

```
Step 1: S3 と同じ（Flash が統合プロンプト生成）
Step 2: Pro が [キャラ参照] + [seed] + [背景参照] + 統合プロンプト で生成
```

**検証ポイント**: seed を Pro に渡すことで構図保持が向上するか。S3 との差分。

### S3m: Flash統合分析（最小指示メタプロンプト） → Pro生成

S3 のメタプロンプトを A-2 知見の**最小指示原則（M1）**に準拠させたバリエーション。
A-2 では分析項目を列挙するメタプロンプト（M2〜M4）より、最小指示で Flash に分析を委ねる M1 が最良だった。
S3 のメタプロンプトは分析項目を列挙しており M2 寄りのため、最小指示版との比較で効果を検証する。

```
Step 1: Flash が [seed] + [背景参照] を分析 → 統合シーン記述を生成
  メタプロンプト:
    「Analyze both images carefully.
     Image 1 is the original scene. Image 2 is the target environment.
     Generate an image generation prompt that recreates the scene from image 1
     in the environment from image 2.
     The character should be: {Identity Block}
     The character's pose: {pose_instruction}
     Output only the prompt text, nothing else.」

Step 2: Pro が [キャラ参照] + [背景参照] + 統合プロンプト で生成
```

**検証ポイント**: S3（分析項目列挙型）と比較して、最小指示で Flash の自律的分析に委ねた方が品質が上がるか。A-2 の M1 知見がシーン統合タスクにも適用できるか。

### S5: 精度維持型分割 — シーン先行生成 → キャラ差替

タスクを「環境生成」と「人物配置」に分割する。

```
Step 1: シーン生成（人物なし or 汎用人物）
  入力: [seed] + [背景参照] + テキスト
  「Image 1 の構図で Image 2 の環境を描いた写真を生成。
   人物は元の位置・ポーズのまま保持。」
  → 背景が統合された自然なシーン（元 seed の人物が残る）

Step 2: キャラクター差替（A-1 手法）
  入力: [キャラ参照] + [Step 1 出力] + テキスト
  A-1 の全部入りプロンプト（参照画像 + Identity Block + ALL CAPS）
  → Step 1 のシーン上でキャラクターを差替
```

**検証ポイント**: Step 1 で環境が自然に統合されるか。Step 2 でバーチャル背景感が再発しないか。
**注意**: Step 1 は実質的に A-3（背景変更）と同じ。ただし2画像入力で Identity Block なし。

### S6: 精度維持型分割 — Flash環境分析 → キャラ+環境統合生成

Flash を使って背景参照の環境を分析し、テキストで環境コンテキストを提供。
人物と環境を1回のPro生成で統合するが、Flash の分析で精度を補助。

```
Step 1: Flash が [背景参照] を分析 → 環境記述テキスト
  「この画像の環境を詳細に記述せよ（場所、植生、地面、空、照明、奥行き）」

Step 2: Flash が [seed] を分析 → 構図記述テキスト（A-2 手法の応用）
  「この画像の構図を分析し、人物差替+ポーズ変更のプロンプトを生成」
  ※ A-2 のメタプロンプトに環境コンテキストを追加

Step 3: Pro が [キャラ参照] + [背景参照] + 統合プロンプト で生成
  Step 1 の環境記述 + Step 2 の構図記述を統合したプロンプト
```

**検証ポイント**: Flash の分析を2段階に分けることでプロンプト精度が上がるか。コスト増の価値があるか。

## テキスト除去の扱い

シーン統合生成アプローチでは、seed 画像を直接編集するのではなく新規生成するため、
**テキストオーバーレイは自動的に排除される**（seed のテキストは生成結果に引き継がれない）。

ただし、seed を構図参照として Pro に入力するパターン（S1, S4, S5）では、
モデルがテキストを再現してしまう可能性がある。その場合は最終ステップで A-5 のテキスト除去を適用する。

## 検証設計

### 入力画像

- **キャラクター参照**: `poc/seamless/reference/front.png`
- **背景参照**: `poc/seamless/reference/sakura.jpg`
- **seed 画像**: `seeds/captures/tamachan_life_/4.png`（全身・立ち・外出 — 屋外背景との整合性が最も高い）

seed_4 を主軸にする理由:
- 全身構図で環境統合の評価がしやすい
- 立ちポーズ + 屋外背景の組み合わせが自然（A-3 知見: ポーズと背景の状況的整合性が重要）
- B-1 v1〜v3 および A-3 で比較データが蓄積されている

### 評価軸

| 評価軸 | 重み | 説明 |
|--------|------|------|
| 空間認識・接地感 | 最重要 | 人物が環境に自然に溶け込んでいるか。足元・影・奥行き |
| 人物の一貫性 | 高 | キャラ参照と顔・髪・服装が一致しているか |
| 構図保持 | 高 | seed の構図（カメラアングル、人物の位置）が再現されているか |
| 背景の品質 | 中 | 背景参照の雰囲気（桜・菜の花・川・山）が再現されているか |
| ポーズ反映 | 中 | 指示したポーズ（セルフィー）が実現されているか |
| 前景オブジェクト | 低 | seed のオブジェクト（バッグ等）が保持されているか（生成アプローチでは犠牲になる可能性あり） |

### 実行順序

1回目: **S1, S3, S5** を seed_4 で実行（方針A, B, C の代表パターン各1つ）
→ 方向性の有効性を確認

2回目: 1回目で有望なパターンの**派生パターン**を実行
→ 例: S1 が良ければ S2 を追加、S3 が良ければ S4, S3m を追加

3回目: 最良パターンを**全3 seed** で実行
→ 構図バリエーションでの安定性を確認

### コスト見積もり

| 回 | パターン数 | seed数 | API呼び出し/パターン | 合計呼び出し | 推定コスト |
|----|-----------|--------|---------------------|-------------|-----------|
| 1回目 | 3 | 1 | 1〜2 | 4〜6 | $0.16〜$0.24 |
| 2回目 | 2〜3 | 1 | 1〜2 | 2〜6 | $0.08〜$0.24 |
| 3回目 | 1〜2 | 3 | 1〜2 | 3〜6 | $0.12〜$0.24 |
| **合計** | | | | **9〜18** | **$0.36〜$0.72** |

## 前提条件

- **Python SDK（`google-genai`）を使用すること**（REST API は非推奨）
- REST API では `aspect_ratio` を制御できず、参照画像使用時にアスペクト比が崩壊する（A-1 知見）
- 出力アスペクト比は `ImageConfig(aspect_ratio="9:16")` で明示指定する
