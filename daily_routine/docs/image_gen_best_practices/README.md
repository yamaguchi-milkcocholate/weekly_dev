# 画像生成ベストプラクティス

PoC 検証から得られた、画像生成タスクごとのベストプラクティスを蓄積するナレッジベース。

## 背景

画像生成 API（Gemini, FLUX Kontext, Runway 等）は、タスクの種類（人物差し替え、ポーズ変更、背景変更など）によって最適な手法が異なる。PoC 検証を通じて帰納的に得られた知見をここに蓄積し、本番パイプラインの設計・実装に活用する。

## ディレクトリ構成

```
docs/image_gen_best_practices/
├── README.md              # 本ファイル（運用ルール）
└── {タスク名}.md           # タスク別ベストプラクティス
```

### ファイル命名規則

- タスク単位で1ファイル（例: `character_swap.md`, `pose_change.md`, `background_change.md`）
- ファイル名はスネークケース、英語

## 各ファイルの記載フォーマット

```markdown
# {タスク名}

## 推奨手法

- **推奨 API**:
- **方式**: （1パス / マルチパス / etc.）
- **コスト**:

## プロンプト例

（実際に高品質な結果を出したプロンプトを記載）

## 注意点・アンチパターン

（やってはいけないこと、品質が下がるパターン）

## 検証エビデンス

- 検証レポート: `poc/xxx/reports/xxx.md`
- 生成サンプル: `poc/xxx/generated/xxx.png`
```

## 運用ルール

1. **PoC 検証で知見が確定したら追記する**: 検証レポート(`poc/*/reports/`)で結論が出たタスクについて、ベストプラクティスを抽出して記載する
2. **検証エビデンスを必ずリンクする**: 根拠となる PoC レポートと生成サンプルへのパスを記載し、追跡可能にする
3. **知見が更新されたら上書きする**: 新たな検証で従来の結論が覆った場合、古い内容を更新し、変更理由を記載する
4. **汎用的な知見は横断的にまとめる**: 特定タスクに限らない知見（例: 「1パスが最良」「参照画像追加は逆効果になりうる」）は別途横断ファイルにまとめてもよい

## 現在確立されている横断的知見

Phase 1〜4 および Phase A-1〜A-3 の検証から得られた汎用知見:

1. **1パスが最良**: 複数の変更（人物差し替え + ポーズ変更等）は1パスで同時指示する方が品質が高い
2. **マルチパスは劣化する**: パスを重ねるほど画質が劣化し、意図した調整効果が薄れる
3. **参照画像は SDK で正しく使えば有効**: Phase 1〜4 で「逆効果」だったのは REST API の制限が原因。Python SDK（`google-genai`）+ `ImageConfig(aspect_ratio=)` で解決
4. **Gemini 3 Pro は汎用性が高い**: ポーズ変更・人物差し替え・環境変更の全てで高品質かつ最安（$0.04/枚）
5. **Python SDK（`google-genai`）を使用する**: REST API では `aspect_ratio` を制御できない
6. **タスクによって最適手法が正反対になる**: 人物差し替えでは Identity Block + ALL CAPS が必須だが、背景変更では逆効果。タスクごとのベストプラクティスを参照すること
7. **モデルの動作モード（編集 vs 生成）を意識する**: `Replace X` は編集モード、`Place into` / Identity Block は生成モードを誘発する。空間認識は編集モードが優れる
8. **1パス vs 2パス（Flash分析→Pro生成）は入力画像数で判断する**: 入力画像1枚なら1パス直接指示、2枚以上の融合なら Flash 分析を挟む2パスが安定（C1 検証）
9. **Flash 記述はマルチアングル一貫性のアンカーになる**: 同一テキスト記述を複数アングルで共有することで、顔・体型・服装の統一を担保（C1-F2-MA 検証）
10. **Identity Block は生成後の画像から抽出する**: 生成前の指示テキストではなく、実際に生成された画像を分析した記述を使う。意図と結果の乖離を防ぐ（C1-ID 検証）
11. **全身条件は `from head to feet` + `space below the feet` で明示する**: `Full body shot` だけでは足元が切れるリスクがある（C1-F2-MA 検証）
12. **環境再現でも参照画像の直接入力が最適**: 人物入り参照写真から環境のみを再現する場合、参照画像を直接 Pro に渡す1パス（C2-R2/R3）が最も忠実。Flash テキスト化を挟むと細部が失われる（C2 検証）
13. **環境記述テキストは生成画像から自動抽出する**: C2-ED で Flash 分析すれば場所・季節・時間帯・照明・色彩・雰囲気を正確にカバー。C3 での環境再現アンカーとして機能（C2-ED 検証）
14. **シーン融合（キャラ×環境）では最小指示 Flash 分析が最良**: Flash には画像 + Identity Block + ポーズのみ渡す。環境記述テキスト・シナリオコンテキストの追加は Flash プロンプトを肥大化させ、Pro の過制約を引き起こす（動的ポーズで接地感低下）。B-1 S3M の最小指示知見と一致（C3-I1 > C3-I2 検証）
15. **環境画像が構造的忠実さに重要**: テキストのみでは環境の概念は合うが構造物・構図が異なる。参照環境を忠実に再現するには環境画像の入力が必須（C3 検証）
16. **ポーズ指示がカメラ構図にも影響する**: selfie ポーズでは Pro が自動的に自撮り風近接アングルに調整する。全身が必要な場合は `full body from head to feet` を明示的に追加する（C3 検証）
17. **selfie は `taking a selfie` と簡潔に書く**: `holding up a smartphone` とデバイスを明示すると Pro が第三者視点（スマホを持つ人を外から撮影）に切り替わる。デバイスを書かず `taking a selfie` とだけ指示すれば、内カメラ視点（カメラ＝スマホ）の自然な自撮り写真になる（C3 selfie 検証）
