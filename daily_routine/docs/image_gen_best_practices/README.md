# 画像生成ベストプラクティス

PoC 検証から得られた、画像生成タスクごとのベストプラクティスを蓄積するナレッジベース。

## 背景

画像生成 API（Gemini, FLUX Kontext, Runway 等）は、タスクの種類（人物差し替え、ポーズ変更、背景変更など）によって最適な手法が異なる。PoC 検証を通じて帰納的に得られた知見をここに蓄積し、本番パイプラインの設計・実装に活用する。

## ディレクトリ構成

```
docs/image_gen_best_practices/
├── README.md              # 本ファイル（運用ルール）
└── {タスク名}.md           # タスク別ベストプラクティス
```

### ファイル命名規則

- タスク単位で1ファイル（例: `character_swap.md`, `pose_change.md`, `background_change.md`）
- ファイル名はスネークケース、英語

## 各ファイルの記載フォーマット

```markdown
# {タスク名}

## 推奨手法

- **推奨 API**:
- **方式**: （1パス / マルチパス / etc.）
- **コスト**:

## プロンプト例

（実際に高品質な結果を出したプロンプトを記載）

## 注意点・アンチパターン

（やってはいけないこと、品質が下がるパターン）

## 検証エビデンス

- 検証レポート: `poc/xxx/reports/xxx.md`
- 生成サンプル: `poc/xxx/generated/xxx.png`
```

## 運用ルール

1. **PoC 検証で知見が確定したら追記する**: 検証レポート(`poc/*/reports/`)で結論が出たタスクについて、ベストプラクティスを抽出して記載する
2. **検証エビデンスを必ずリンクする**: 根拠となる PoC レポートと生成サンプルへのパスを記載し、追跡可能にする
3. **知見が更新されたら上書きする**: 新たな検証で従来の結論が覆った場合、古い内容を更新し、変更理由を記載する
4. **汎用的な知見は横断的にまとめる**: 特定タスクに限らない知見（例: 「1パスが最良」「参照画像追加は逆効果になりうる」）は別途横断ファイルにまとめてもよい

## 現在確立されている横断的知見

Phase 1〜4 および Phase A-1〜A-3 の検証から得られた汎用知見:

1. **1パスが最良**: 複数の変更（人物差し替え + ポーズ変更等）は1パスで同時指示する方が品質が高い
2. **マルチパスは劣化する**: パスを重ねるほど画質が劣化し、意図した調整効果が薄れる
3. **参照画像は SDK で正しく使えば有効**: Phase 1〜4 で「逆効果」だったのは REST API の制限が原因。Python SDK（`google-genai`）+ `ImageConfig(aspect_ratio=)` で解決
4. **Gemini 3 Pro は汎用性が高い**: ポーズ変更・人物差し替え・環境変更の全てで高品質かつ最安（$0.04/枚）
5. **Python SDK（`google-genai`）を使用する**: REST API では `aspect_ratio` を制御できない
6. **タスクによって最適手法が正反対になる**: 人物差し替えでは Identity Block + ALL CAPS が必須だが、背景変更では逆効果。タスクごとのベストプラクティスを参照すること
7. **モデルの動作モード（編集 vs 生成）を意識する**: `Replace X` は編集モード、`Place into` / Identity Block は生成モードを誘発する。空間認識は編集モードが優れる
