# PoC Step 6: 3Dレンダリング結果とユーザーのイメージ画像からテクスチャ・雰囲気が反映されたリアルな部屋映像を生成できる

> [ロードマップに戻る](./roadmap.md)

## 検証目標

Step 5のレンダリング画像（テクスチャなし）とユーザーのイメージ画像（1枚）を入力として、イメージの雰囲気・テクスチャが反映されたリアルな静止画像を生成できる。

---

## このステップの役割

```
Step 5の出力（レンダリング画像・テクスチャなし）
    ×
ユーザーのイメージ画像（雰囲気の基準・1枚）
    ↓
AIがテクスチャ・雰囲気を合成
    ↓
リアルな部屋の静止画像を出力
```

このステップがパイプラインの最終出力であり、ユーザーが「いいな」と思う雰囲気を実現する核心部分。

## 拡張方針（PoCスコープ外）

PoCではイメージ画像1枚のみを検証する。その後以下の順で拡張していく。

```
A. イメージ画像1枚（PoCスコープ）
    ↓
B. 複数枚のミックス
    ↓
C. 複数枚 + 詳細な指示
```

---

## 入力

| 入力                   | 内容                                                                           |
| ---------------------- | ------------------------------------------------------------------------------ |
| レンダリング画像       | Step 5の出力。空間の形状・アセット配置が正確に反映されているが、テクスチャなし |
| ユーザーのイメージ画像 | 1枚。雰囲気・テクスチャの基準となる画像（別部屋の写真・インテリア雑誌など）    |

### A. ControlNet + IP-Adapter（OSS / ローカル実行）

**概要**: ControlNetで3Dレンダリングの空間構造を維持し、IP-Adapterで参照画像からスタイル・テクスチャを転写する組み合わせ。本ユースケースの最も直接的な解法。

| 項目 | 詳細 |
|------|------|
| 入力 | 3Dレンダリングから出力したDepth Map / Normal Map + スタイル参照画像 + テキストプロンプト |
| 出力 | 構造を維持しつつスタイルが適用された画像 |
| 空間構造維持 | ControlNet Depthで大まかな構造を維持。Normal Mapでより細部の凹凸を保持可能 |
| スタイル転写 | IP-Adapter Plusモデルがパッチ埋め込みで詳細なスタイル情報を捕捉。weightで強度調整可能 |
| 実装 | ComfyUIで比較的容易に構築可能。GPU VRAM 8GB以上推奨 |

**対応モデル別の状況**:

| ベースモデル | ControlNet | IP-Adapter | 成熟度 |
|-------------|-----------|-----------|--------|
| SD 1.5 | Depth, Normal, Canny等すべて利用可能 | 全モデル対応（Plus, FaceID等） | 最も成熟 |
| SDXL | サードパーティ製で対応。Union SDXLも存在 | Plus SDXL ViT-H対応 | 成熟 |
| SD 3.5 Large | 公式でBlur, Canny, Depthの3種のみ。**Normal Map未提供** | InstantX製のみ（発展途上） | 発展途上 |
| Flux | Depth, Canny対応（後述のFlux APIで利用可） | Flux対応版あり | 成熟しつつある |

**強み**: 3DソフトからDepth/Normal Mapを直接エクスポートできるため、推定モデルを介さず高精度な条件マップを得られる。

**制限**: IP-Adapter単体では構造維持不可（ControlNet併用必須）。コンテンツリーク（参照画像の内容が意図せず混入）に注意。

---

### B. InstantStyle / InstantStyle-Plus

**概要**: IP-Adapterベースでスタイルとコンテンツの分離（disentanglement）を改善したフレームワーク。ICLR 2025関連。

| 項目 | 詳細 |
|------|------|
| 入力 | コンテンツ画像 + スタイル参照画像 |
| 出力 | スタイル適用済み画像 |
| 空間構造維持 | Plus版はTile ControlNet + 反転潜在ノイズで元画像のレイアウトを保持 |
| スタイル転写 | スタイル担当レイヤー（第6ブロック）にのみ画像特徴を注入。コンテンツリークを抑制 |
| 実装 | プラグアンドプレイ。追加学習不要 |

**強み**: コンテンツとスタイルの分離が明示的。コンテンツリーク軽減。

**制限**: スタイル精度低下のトレードオフあり。参照画像とプロンプトの意味的ギャップが大きい場合に精度低下。

---

### C. T2I-Adapter / Uni-ControlNet / ControlNet++

**概要**: ControlNetの軽量代替・後継技術群。

| 技術 | 特徴 | 構造維持 | 実用性 |
|------|------|---------|--------|
| **T2I-Adapter** | 約77Mパラメータ（~300MB）と軽量。UNetに重み挿入のみ | ControlNet同等 | SDXL対応済。推論が高速 |
| **Uni-ControlNet** (NeurIPS 2023) | 複数条件（Depth + Edge + Seg等）を単一モデルで同時利用 | 複数条件組み合わせで高精度 | SD1.5ベース。SDXL/Flux対応は限定的 |
| **ControlNet++** (ECCV 2024) | ピクセルレベルのサイクル一貫性最適化 | mIoU +11.1%, SSIM +13.4%改善 | OSS公開済。報酬モデルの事前訓練が必要 |

---

### D. Flux ControlNet API（Black Forest Labs）

**概要**: FLUX.1/FLUX.2シリーズのControlNet（Depth, Canny）で構造を厳密に維持しながら新しいスタイルの画像を生成。**技術的に最も柔軟で精度の高い商用選択肢**。

| 項目 | 詳細 |
|------|------|
| API利用 | BFL公式API: FLUX.2 Pro $0.055/枚、FLUX 1.1 Pro $0.04/枚。fal.ai / Replicate経由でも利用可 |
| 空間構造維持 | Flux Depthで3D構造・オブジェクト配置を高精度で維持。Blender等の深度パスを直接入力可 |
| スタイル転写 | テキストプロンプト + LoRA でカスタムスタイル適用。IP-Adapter対応（参照画像転写） |
| パイプライン統合 | REST API経由で統合容易。diffusersライブラリでも利用可能 |
| ControlNet Union Pro 2.0 | Canny, Depth, Soft Edge, Pose, Grayscaleの同時条件付けが可能 |

**強み**: 構造維持精度が最も高い。API統合が容易。LoRA/IP-Adapterとの組み合わせでスタイル転写も可能。

**制限**: ControlNet + LoRA等の組み合わせ設定がやや複雑。

---

### E. Adobe Firefly（Structure Reference + Style Reference）

**概要**: Structure ReferenceとStyle Referenceを組み合わせ可能。**3Dグレーレンダーからの変換に最も直接的に対応する設計**。

| 項目 | 詳細 |
|------|------|
| API利用 | Firefly Services API: ~$0.02〜$0.12/枚 |
| 空間構造維持 | Structure Referenceで画像のアウトライン・深度情報を明示的に抽出・適用。strength 1〜100で調整可 |
| スタイル転写 | Style Referenceで参照画像の色彩・テクスチャ・ムードを転写 |
| パイプライン統合 | REST API経由。Adobe Developer Portal にドキュメント充実 |
| 商用安全性 | 著作権侵害リスクが低い（学習データが権利クリア） |

**強み**: 前処理（深度マップ生成）不要で3Dレンダーをそのまま入力可能。Structure + Styleの組み合わせが直感的。

**制限**: Flux/Midjourneyと比較して生成品質がやや劣るとの評価あり。

---

### F. GPT Image 1.5（OpenAI）

**概要**: GPT-4oに統合された画像生成・編集エンジン。参照画像を渡してスタイル転写・リテクスチャが可能。

| 項目 | 詳細 |
|------|------|
| API利用 | `gpt-image-1`: low $0.01/枚、medium $0.04/枚、high $0.17/枚 |
| 空間構造維持 | 1.5で大幅改善。ただしControlNet的な明示的構造制約なし。モデルの理解力に依存 |
| スタイル転写 | マルチモーダルモデルのためコンテキスト理解が深い。スタイル転写品質は高い |
| パイプライン統合 | REST API + Python SDK（openai）で最も簡単に統合可能 |

**強み**: API統合が最も簡単。マルチモーダル理解による高品質なスタイル転写。

**制限**: 明示的な深度マップ/構造制約がない。構造維持はプロンプト設計に依存。

### H. 3Dメッシュテクスチャ生成（3Dモデルを保持している場合）

レンダリング後の2D画像ではなく、3Dメッシュに直接テクスチャを生成する手法。

| 技術 | 概要 | 速度 | 品質 |
|------|------|------|------|
| **Text2Tex** (ICCV 2023) | 深度認識拡散モデル + インペインティングで段階的テクスチャ合成 | 1分以上 | 高品質 |
| **Paint3D** (CVPR 2024) | 照明なし2K UVテクスチャマップ生成 | 1分以上 | 高解像度 |
| **Meta 3D TextureGen** (2024) | 法線/位置マップ条件付け。4Kアップスケール対応 | ~19秒 | 全ベースライン上回る |
| **Make-A-Texture** (2024) | 最速手法。バックプロジェクション + バイリニアスプラッティング | A100で4.56秒 | Paint3D同等 |

**強み**: メッシュに直接テクスチャを適用するため、マルチビューの一貫性が高い。

**制限**: PoCの入力は「レンダリング済み2D画像」のため、メッシュデータが必要。パイプラインの前段（Step 5）との連携設計が必要。

### I. IC-Light V2（照明制御特化）

**概要**: ControlNet作者lllyasvielによる照明制御特化モデル。ICLR 2025でフルスコア（10,10,10,10,10）獲得。

| 項目 | 詳細 |
|------|------|
| 入力 | 画像 + 照明条件（テキスト or 光源画像） |
| 出力 | リライティング済み画像 |
| 構造維持 | 画像の構造・コンテンツを維持しつつ照明のみを変更 |
| 実用性 | V2はFluxベースで高解像度ネイティブ対応 |

**強み**: 他手法（ControlNet + IP-Adapter等）と組み合わせて、最終段で照明を調整する用途に最適。

**制限**: 照明制御に特化。スタイル全般の転写には別手法が必要。

### PoC推奨アプローチ

**第一優先: ControlNet (Depth) + IP-Adapter (Plus) on SDXL**
- 3DソフトからDepth Mapを直接エクスポートし、参照画像でスタイルを指定
- SDXLはControlNet・IP-Adapterともにエコシステムが最も成熟
- ローカル実行でイテレーションが高速

**第二優先: Flux ControlNet (Depth) API**
- BFL公式API or fal.ai経由で利用
- 構造維持精度が最も高く、API統合も容易
- クラウド実行でGPU不要

**第三優先: Adobe Firefly (Structure + Style Reference)**
- 前処理不要で3Dレンダーをそのまま入力可能
- 商用安全性が高い
