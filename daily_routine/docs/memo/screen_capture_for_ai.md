# Claude Codeが画面を自動参照する仕組み

日付: 2026-04-13

## 背景: AIコーディングにおける3DCGの壁

プログラミングとAIコーディングの相性は良い。コードはテキストであり、テスト結果やエラーメッセージで品質を自動判定できる。Claude Codeがコードを書き、テストで検証し、修正するループが成立する。

一方、3DCGでは同じループが成立しにくい。最大の理由は**品質判定が人間の視覚判断に依存する**こと。

| 観点           | プログラミング     | 3DCG                             |
| -------------- | ------------------ | -------------------------------- |
| 品質判定       | テスト通過（自動） | 「リアルに見えるか」（人間の目） |
| デバッグ       | エラーメッセージ   | 「なんか変」                     |
| イテレーション | ミリ秒〜秒         | 秒〜時間                         |

Claude Codeがbpyスクリプトで3Dシーンを構築しても、その結果が「良いか悪いか」を判断する手段がなかった。

## 問題: 画像の受け渡しが高コスト

従来の方法:

1. ユーザーがBlenderの画面を手動でスクリーンショット
2. 画像ファイルとして保存
3. Claude Codeに渡す

これは毎回の手間が大きく、自動化パイプラインの中に組み込めない。bpyスクリプトで地形を生成→マテリアルを適用→ライティングを設定、という一連の流れの中で、各ステップの結果をClaude Codeが確認できない。

## 着想: macOSの画面をClaude Codeが直接見る

macOSの`screencapture`コマンドをClaude Codeが自発的に実行し、ユーザーが見ている画面をそのまま参照する。

この着想のポイントは**Blenderに限定しない**こと:

- Blenderの操作画面（ビューポート、ノードエディタ、プロパティパネル）
- Gaea等の他の3DCGツールの画面
- Webブラウザ（アセットの選定、リファレンス画像の確認）
- エラーダイアログ
- その他あらゆるGUIアプリ

つまり**Claude Codeに「目」を与える**仕組み。

## 実装

### コマンド

```bash
screencapture -x /tmp/claude_screen.png
```

`-x`はシャッター音を無効にする。Claude Codeがこれを`Bash`ツールで実行し、`Read`ツールで画像を読み取る。

### 前提条件

macOSの「画面収録」権限が必要:

- システム設定 → プライバシーとセキュリティ → 画面収録
- Claude Codeを実行しているターミナルアプリに権限を付与

権限がない場合は`could not create image from display`エラーになる。

### スキル化

`~/.claude/skills/screen-capture/SKILL.md`としてグローバルスキルに登録。全プロジェクト・全セッションで利用可能。「画面を見て」等のリクエストで発動する。

## AIが代替できる視覚判断

画面キャプチャにより、以下の判断をClaude Codeが自動で行える:

| 判断                                          | 精度                                     |
| --------------------------------------------- | ---------------------------------------- |
| テクスチャが適用されているか                  | 高                                       |
| ライティングが設定されているか                | 高                                       |
| 植生が散布されているか                        | 高                                       |
| スケールが明らかにおかしい                    | 高                                       |
| レンダリングが破綻している                    | 高                                       |
| GUIの操作状態（どのメニューが開いているか等） | 高                                       |
| 「リアルに見えるか」                          | 中（リファレンス画像との比較で精度向上） |
| 構図・雰囲気が意図通りか                      | 低（意図は人間の頭の中）                 |

## 活用パターン

### パターン1: 対話的な操作ガイド

ユーザーがGUIアプリで迷ったとき、Claude Codeが画面を見て現在の状態を把握し、次の操作を案内する。

### パターン2: 自動化パイプラインの中間チェック

```
bpyスクリプト実行 → screencapture → Claude Codeが品質チェック
  → 問題あり → パラメータ調整して再実行
  → 問題なし → 次のステップへ
```

### パターン3: エラー診断

GUIアプリのエラーダイアログやBlenderのビューポートの異常をClaude Codeが画面から直接読み取り、原因を診断する。

## 意義

プログラミングと3DCGのAI自動化における最大の差（品質判定の自動化）を部分的に解消する仕組み。完全な代替ではないが、基本的なチェックをAIが行い、人間は最終的な美的判断のみに集中できる。

3DCGに限らず、**GUIベースのあらゆるアプリケーション操作**でClaude Codeが視覚的なコンテキストを持てるようになる汎用的な仕組み。
