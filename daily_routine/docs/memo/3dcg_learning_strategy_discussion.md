# 3DCG学習戦略の議論と方針転換

日付: 2026-04-13

## 発端

Blenderで自然環境の3DCG制作を学習中。素人レベルのアウトプット（地形+海+植生プレースホルダー）は作れるようになったが、**ここからどうやって実務レベルに到達するのか、今のやり方で合っているのか**という不安があった。

## 議論の経過

### 1. 収益化の見通しが立たない

最初の問いは「収益化の見通しが立たない」だった。議論を通じて、以下が明確になった:

- Blender単体のスキルで稼ぐ（3DCGフリーランス）のではない
- **フィジカルAI時代に3DCGを知っている人が重宝されるという予測**に基づく先行投資
- Blenderの操作をAIで自動化し、素人の自分でも十分なアウトプットを出せる環境を作る
- そのアウトプットで実務問題を解決して収益を生む

### 2. 「素人 + AI = 変なもの」との矛盾？

以前の実験で「素人 + AI = 変なものしか作れない」と結論していた。しかしこれは修正された:

- **羅針盤の問題**はデータソース（公式マニュアル等）を渡すことで解決の目処がある
- **全工程の自動化**はまだ無理。人間の判断が必要な工程がある
- **前提知識**がないと3DCGを扱えないのも事実 → だから今学習している

つまり「素人のままAIで自動化」ではなく、**「3DCGの前提知識を持った人間 + データソース付きClaude Code = 実務レベルのアウトプット」**という構想。

### 3. 学習方法への不安

不安の正体は「素人レベルからリアルな再現ができるレベルに到達する方法が、今のやり方で合っているのか」だった。具体的には:

- 単にサンプルをいじっているだけではないか
- もっといいツールややり方があるのではないか

### 4. 調査の実施

3つの調査を並列で実行:

1. **プロの環境アーティストのワークフロー** — ツールの組み合わせ、作業順序
2. **環境CG向けツール・アセット** — Gaea, Megascans等の選択肢と価格
3. **AI × 3DCG環境生成の最新事例** — 2024-2026年の最新動向

### 5. 調査結果: 今のやり方は遠回りだった

**プロはBlender単体で完結しない。** 地形は専用ツール（Gaea等）、アセットはフォトスキャンライブラリ（Poly Haven等）を使うのが業界標準。

| 工程         | 今のやり方                   | プロの標準                                            |
| ------------ | ---------------------------- | ----------------------------------------------------- |
| 地形生成     | ANT Landscape + 手動Sculpt   | Gaea / World Creator（侵食シミュレーション付き）      |
| アセット     | 白い球体（プレースホルダー） | Megascans / Poly Haven（フォトスキャンアセット）      |
| マテリアル   | 単色の緑                     | PBRテクスチャ（フォトスキャン or Substance Designer） |
| 植生散布     | Geometry Nodes基礎           | Geo-Scatter / GScatter + 植生アセット                 |
| ライティング | ビューポートデフォルト       | Nishita Sky + HDRI + Sun Light                        |

今のやり方の問題点:

- **全部を手で一から作ろうとしている** — プロはフォトスキャンアセットを積極活用。全部自作はプロもやらない
- **地形生成をBlender内で完結させようとしている** — 専用ツールの侵食シミュレーションはBlender内では再現不可能
- **品質を決める要素（ライティング・マテリアル・アセット）が後回し** — 形よりも光と質感が品質を決める

### 6. 構造的な気づき: プログラミングと同じ

**「一から作るのではなく組み合わせて作る」はプログラミングでも同じ。**

```
プログラミング: ライブラリ + ツール + AIコーディング = ソフトウェア
3DCG:          アセット + ツール + AIコーディング = 3Dシーン
```

いかに自分の開発環境を整えることができるか。そのコアはAIコーディング。

### 7. ゴール構造の整理

```
新ゴール（手段）: AI × 3DCG開発環境を構築する
    ↓ これができれば
旧ゴール（目的）: その環境を使って、自分がリアルな3DCGを作れる
```

- 開発環境そのものが自分のスキルギャップを埋める装置になる
- 手動スキル10年分を、環境構築力 + AIで圧縮する
- **Blender学習の目的は3DCGアーティストになることではなく、ツールチェーンの中身を理解し、AIへの指示精度を上げ、アウトプットの品質を評価できるようになること**

### 8. プログラミングと3DCGの差

構造は同じだが、AIに任せきれる割合がプログラミングより低い:

| 観点               | プログラミング   | 3DCG           |
| ------------------ | ---------------- | -------------- |
| 品質判定           | 自動テスト可能   | 人間の視覚判断 |
| イテレーション速度 | ミリ秒〜秒       | 秒〜時間       |
| 部品の組み合わせ   | 繋げば動く       | 調整が必要     |
| デバッグ           | エラーメッセージ | 「なんか変」   |

プログラミングなら7-8割をAIに任せられる場面でも、3DCGでは5-6割程度。人間の評価ループを効率的に回す設計がプログラミング以上に重要。

### 9. 解決策: 画面キャプチャによるAIの「目」

3DCGの品質判定が人間の視覚判断に依存する問題に対して、**macOSのscreencaptureコマンドでClaude Codeが画面を直接参照する仕組み**を構築した。

```bash
screencapture -x /tmp/claude_screen.png  # Claude Codeが自発的に実行
```

これにより:

- 手動でスクリーンショットを保存・共有する手間がなくなる
- bpyスクリプト実行後にBlender画面を自動確認して品質チェックできる
- Blender以外のアプリ（Gaea等）の操作画面も確認できる
- 3DCG開発環境の「目」として機能する

AIが代替できる判断（テクスチャ適用の有無、ライティングの設定、スケールの異常等）を自動化し、人間は最終的な美的判断のみに集中できる。

このスキルは `~/.claude/skills/screen-capture/SKILL.md` として全プロジェクトで利用可能にした。
