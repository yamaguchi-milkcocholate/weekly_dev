## 各PoCの結論

### PoC 0: floor-plan-to-elements 統合パイプライン — 完了
- 3つの旧スキル（floor-plan-filter, floor-plan-walls, floor-plan-to-walls）を2スキル＋1オーケストレータに統合
- Phase 1（floor-plan-trace）: potrace動的閾値選択（50/60/70/80%比較）→ 視覚的フィルタリング → クリーンSVG
- Phase 2（floor-plan-elements）: PNG化 → Zhang-Suen法スケルトン化 → ピクセルスキャン → 壁・柱rectの反復配置（サブエージェントレビュー）
- 検証結果: `madori_1ldk_1.png`（1LDK）→ 壁44個 + 柱7個を自動配置。外壁・内壁・柱の位置精度良好
- PoC 1で課題だった「間取りPNG→壁rect SVG」をスキルとして自動化
- 詳細: `poc/3dcg_poc0_a/memo.md`

### PoC 1: 3D空間データ構築
- Gemini APIは日本語間取り図の認識精度が不十分
- Blender手動トレース + MASt3Rでのテクスチャ補完に方針転換
- 間取りPNG→壁rect SVGはClaude Codeの視覚認識+手作業調整で実用レベル → **PoC 0でスキル化により自動化**

### PoC 2: 画像→3Dアセット抽出
- Tripo AI APIでEC画像→GLB変換に成功（5アセット検証済）
- 出力: `poc/3dcg_poc2/output/` にGLBファイル（bed, desk, chair, counter, closet, dining_table）

### PoC 3: レイアウト提案
- Claude Codeで間取り画像+寸法JSON+アセットBBoxから配置座標を生成
- 幾何学的な衝突回避はOK、ただし**生活動線の考慮が弱い**（空間充填優先）
- 出力: `poc/3dcg_poc3/output/layout_proposal.json`（10アセットの配置座標・向き）

### PoC 4: 3D空間へのアセット配置 — 完了
- Blender+bpyでGLBアセットの自動配置を実現
- front方向の自律判定（スケール比率による検証）を含む配置パイプラインを構築
- 3回のPatchで接地不良・scale歪み・front方向誤判定を解消
- `blender-placement` スキルとして汎用化済み
- 成果物: `poc/3dcg_poc4/output/scene.blend`（配置済みシーン）

### PoC 5: 任意カメラ位置レンダリング — 完了
- EEVEE + HDRI環境照明（Material Preview相当）方式を採用
- Workbench（グレー単色で判別困難）、Cycles+ライト（壁法線が外向きで真っ暗）を試行して不採用
- マテリアルなしオブジェクトへの一時マテリアル付与、俯瞰カメラ時の天井自動非表示を実装
- `blender-camera-render` スキルとして汎用化済み
- 成果物: `poc/3dcg_poc5/output/renders/`（複数カメラアングルのレンダリング画像）

### PoC 6: テクスチャ/雰囲気適用 — 完了
- **採用方式**: Gemini 3.0 Pro Image + テキストスタイル方式
- スタイル参照画像をClaude Codeがテキスト記述に変換し、レンダリング画像1枚+テキストプロンプトでGeminiに生成させる
- Runway・LUMAはテキストスタイル方式でも構造が崩壊。Geminiが構造維持で圧倒的に優位
- 2画像入力（レンダリング+スタイル画像）はモデルがスタイル画像の空間的性質を取り込んでしまい不安定
- プロンプト設計の知見: オブジェクト追加禁止の明示的ルール、スタイルテキストからオブジェクト名を除去、"re-skinning"メタファーが有効
- 最終スコア: 構造維持97.8、スタイル反映95.8、生成品質96.8、一貫性90
- コスト: $0.134/枚（Gemini 3.0 Pro Image）、6カメラで約$0.80
- `3dcg-style-apply` スキルとして汎用化済み

### PoC 8: Geminiマルチアングル一貫性（小物追加） — 完了（及第点）
- **採用方式**: 俯瞰画像に小物を自由追加 → 俯瞰出力をreferenceとして各カメラに小物追加
- 俯瞰1回 + カメラ5回で合計6枚生成。再生成を含め合計9回（$1.21）
- **画角維持の課題**: referenceの俯瞰画像に引きずられてカメラアングルが俯瞰に変わる問題が頻発。extra-instructionsで元画像の構図を詳細に記述することで回避可能だが、再生成が必要になるケースが多い
- **小物一貫性の評価**: 俯瞰画像の小物が各カメラで完全一致はしない。種類（植物・カップ・本等）は概ね一致するが、正確な位置・個数は揃わない。SNS用途には及第点
- **結論**: Geminiのマルチアングル一貫性は「おおよそ一致」レベル。完全一致が必要な場合は3DCG側で小物を配置しre-skinningのみに限定する方針が確実
- `3dcg-style-apply-living` スキルとして汎用化済み

### PoC 9: V2Vスタイル転写の生活感検証 — 完了
- **目的**: Blenderカメラパス動画 → V2Vスタイル転写で、フォトリアルな生活感のある動画が作れるか検証
- **検証内容**: カメラ1→カメラ2の5秒ウォークスルー動画（EEVEE+HDRIレンダリング）を3サービスでV2V
- **比較結果**:

| サービス | モデル | 構造維持 | フォトリアル感 | 生活感 | 解像度 | コスト/5秒 | API最新モデル |
|---------|--------|---------|-------------|-------|--------|----------|-------------|
| Luma | ray-2 adhere_1 | ◎ | ○ | △ 控えめ | 720p | ~$1.75 | × ray3.14未対応 |
| Runway | gen4_aleph | ○ | ○ | ○ | 720p | ~$0.75 | △ V2V専用止まり |
| **Kling** | **V3 Omni** | ○ | **◎** | **○** | **1080p** | ~$0.42〜$0.84 | **◎ 最新モデル対応** |

- **採用**: **Kling V3 Omni**
  - 最新モデルがAPI経由で利用可能（自動化可能）
  - 1080p出力、コストも中程度
  - 初回は天井なし（青空）問題があったが、プロンプト調整で解決
  - 「real estate listing video」「shot on mirrorless camera」等のフォトリアル指示、天井・照明の明示的記述、素材の質感詳細記述が有効
- **Reference画像の効果**: PoC 8出力（カメラ1・カメラ2のスタイル適用済み画像）をreferenceとして渡すことで品質が大幅向上
  - フローリングがヘリンボーン柄の実写質感に。照明がスポットライトレール+ダウンライトで自然に
  - CG感がほぼ消え、不動産物件の内見動画レベルに到達
  - referenceなしでは「ゲームっぽさ」が残るが、referenceありで実写感に転換
- **残課題**: 壁がBlenderオブジェクトのフラットな質感のまま残りがち。対策として:
  - Blender側で壁にマテリアル/テクスチャを設定する
  - reference画像で壁の質感をより強く指定する
- **Luma/Runway所感**: API経由では最新モデルが使えない（Luma Ray3.14 Modify、Runway Gen-4.5はV2V API非対応）。Klingは公式APIで最新V3 Omniが使える点で優位
- **詳細**: `poc/3dcg_poc9/kling_v3_manual.md`

### PoC 0_b: Blenderリッチ化（PBRマテリアル + 3D構造強化）— 進行中

- **完了**: svg_to_blender.pyにPBRマテリアル実装（Roughnessノイズ付き）
  - Noise Texture → Map Range → Roughness入力で5-10%のRoughness変動を付与
  - 壁(0.7±8%), 柱(0.8±10%), ドア(0.45±6%), 床(0.55±7%), 天井(0.8±5%)
  - 3間取り（1/, 2/, 3/）で生成確認済み（78, 67, 92オブジェクト）
- **完了**: カメラ位置を室内目線（高さ1.5m）に調整
- **未対応（patch_memo.md記載）**:
  1. **窓の開口部作成（最優先）** — Cyclesでは閉じた箱にHDRI外部光だけでは室内に光が届かず真っ黒。窓の開口部がないとCyclesレンダリングが成立しない
  2. **色調整（中優先）** — 床ダークブラウン + 壁暖色系で全体が茶色い。V2V入力にはコントラスト（壁/床/ドアの区別）を優先すべき
  3. **カメラ壁衝突回避（低優先）** — 壁に近すぎるとグレー一色になる。壁から0.3m以上のマージン必要
- 詳細: `poc/3dcg_poc0_b/memo.md`, `poc/3dcg_poc0_b/patch_memo.md`

### PoC 0_c: V2V入力用カメラカット + Cyclesレンダリング — 完了・スキル化済

- **方針**: 連続ウォークスルーではなく、PR動画に合った**5秒独立カット方式**を採用
  - スプライン補間（Catmull-Rom/Hermite）を検証した結果、独立カット方式に変更
  - カメラの進行方向と向きは独立（PR動画では一般的）
- **ワークフロー（3フェーズ）**:
  1. 空間分析+カット設計（俯瞰画像→draw.ioでパス描画→Blenderでカメラ配置・調整→アニメーション確認）
  2. Cyclesフレームレンダリング（GPU 32s, 480×270, smooth_step補間）
  3. ffmpegで30fps mp4変換
- **スキル化**: `v2v-camera-cuts` スキルとして汎用化。render_overhead.pyの汎用化、generate_drawio.pyの新規作成を含む
- **成果**: 4カット × 150フレーム = 600フレーム、fast設定(32s, 480×270)で完了
  - C1: 玄関→廊下ドリーイン（南向き直進）
  - C2: LDK南側を左→右スイープ（北東方向を向きながら移動）
  - C3: LDK右→左トラッキング（カメラ南〜南西向き）
  - C4: LDK南側から斜め俯瞰→全景ズームアウト
- **スキル実行時の知見**:
  - draw.io背景画像が巨大すぎる問題 → `generate_drawio.py`に長辺800pxリサイズ処理を追加
  - C2スイープでrot_z=290°→-0.8°の補間で大回転発生 → Blenderでカメラ調整し-35.6°→-1.6°に修正
- 詳細: `poc/3dcg_poc0_c/memo.md`

### 3DCGシーン強化 ← PoC 0_b で検証中

PoC 9の結果、V2V（Kling V3 Omni + reference画像）で床テクスチャ・照明・小物はAI側でカバーできることが判明。
一方で**Blender側の3D構造の正確さ**がV2V出力品質に直結することも確認。壁がフラットな板のままだとV2Vでもそのまま残る。

**V2Vでカバーできるもの（Blender側の対応不要）:**
- 床テクスチャ → V2Vがリアルな木目フローリングを生成
- 小物・装飾 → V2V + referenceで自然に追加される
- 照明 → V2Vがダウンライト・スポットライト等を自然に生成

**Blender側で強化が必要なもの（V2Vでカバーできない）:**

| 要素 | 現状の問題 | 対応 |
|------|-----------|------|
| 壁・窓・ドアの区別 | 全て同じフラットな板で区別がない | オブジェクトとして分離し、窓枠・ドア枠を明確にする |
| 天井 | 天井なし（レンダリング時に非表示） | 天井オブジェクトを配置する。ないと青空が見えるorV2Vが室外と誤認 |
| 隙間・開口部の忠実さ | 壁の接合部に隙間がある | 壁同士の接合を正確に行い、隙間をなくす |
| 窓の景色 | 白飛び | 背景プレーン（景色画像テクスチャ付き板）を窓奥に配置 |

### PoC 7: ウォークスルー動画生成（3DCGシーン強化後）
