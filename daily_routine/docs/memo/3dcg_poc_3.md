# PoC Step 3: Claude Codeが空間を認識してレイアウトを提案できる

> [ロードマップに戻る](./roadmap.md)

## 検証目標

Claude Codeが3D空間を認識し、空間の広さ・形状を根拠に各アセットの配置座標を提案できる。

---

## このステップの核心

Claude Codeを処理エンジンとする方針において、空間認識は最も基礎的な能力となる。
問いは2つ：

1. **Claude Codeは空間を認識できるか？** — テキスト（JSON）と画像（平面図）のマルチモーダル入力で空間を理解できるか
2. **認識できる情報を作れるか？** — 3Dデータ（.blend）からClaude Codeが理解しやすい表現に変換できるか

テクスチャ・雰囲気はこのステップでは扱わない。後工程「3Dレンダリング結果 × ユーザーのイメージ画像」で適用する。

---

## Claude Codeの空間認識モデル

Claude Codeが空間を認識する手段は2つあり、**両方を組み合わせる**のが最も効果的。

### (1) 平面図画像 — 空間の形状・構造を視覚的に把握

- 部屋の形、壁・ドア・窓の位置関係を直感的に理解できる
- 「ここにソファを置くと動線が塞がる」といった空間的判断が可能
- JSON座標の羅列では伝わらない空間の全体像を一目で伝えられる

### (2) 寸法テキスト（JSON） — 正確な数値で配置座標を決定

- 部屋の幅×奥行（メートル単位）
- 壁・ドア・窓の位置と寸法
- アセットのバウンディングボックスサイズ
- 座標系の定義（右手系、Z-up、メートル単位）

### なぜ両方が必要か

| 入力 | 得意なこと | 苦手なこと |
|------|-----------|-----------|
| 平面図画像のみ | 空間の形状把握、動線判断 | 正確な座標値の出力 |
| 寸法JSONのみ | 正確な座標計算 | 壁パーツ群から空間全体を把握すること |
| **両方** | **形状理解 + 正確な座標出力** | — |

## 3Dデータからの情報変換パイプライン

Claude Codeが認識できる情報を3Dデータから生成する。

```
.blend (3Dデータ)
  │
  ├─→ [Blender bpy: トップダウンレンダリング]
  │     カメラをZ軸上方に配置、正射投影
  │     壁・ドア・窓を色分け描画
  │         │
  │         ▼
  │     floor_plan.png（平面図画像）  ← Claude Codeが視覚認識
  │
  ├─→ [Blender bpy: 空間データ抽出]
  │     居室範囲・開口部を構造化
  │         │
  │         ▼
  │     room_info.json（整理済み寸法）  ← Claude Codeがテキスト認識
  │         {
  │           "room": { "width": 5.2, "depth": 3.5 },
  │           "door": { "position": "south", "center_x": 0.77 },
  │           "window": { "position": "north", "width": 2.0 }
  │         }
  │
  └─→ [trimesh: アセットBBox抽出]
        GLBファイルから実寸を取得
            │
            ▼
        asset_bbox.json  ← Claude Codeがテキスト認識
```

### 変換手法の検討

#### 平面図画像の生成

| 手法 | 概要 | 評価 |
|------|------|------|
| **Blender正射投影レンダリング** | カメラを真上に配置（Z軸）、Orthographicモードでレンダリング。壁・床・開口部をマテリアルで色分け | **採用**: bpyスクリプトで自動化可能。寸法の注釈（テキストオーバーレイ）も追加できる |
| Blender Freestyle SVG出力 | Freestyleレンダラーでエッジをベクター抽出しSVG化 | 設定が複雑。PoC段階では過剰 |
| matplotlib 2D描画 | space_data.jsonの座標値からmatplotlibで2D平面図を描画 | 3Dデータを活かせない。ただしフォールバックとして有用 |

#### 寸法データの構造化

| 手法 | 概要 | 評価 |
|------|------|------|
| **bpyで壁パーツ → 居室単位に集約** | 壁パーツの座標群から居室範囲・ドア・窓の位置を算出し、部屋単位の構造化JSONに変換 | **採用**: 既存の`extract_space_data.py`を拡張。壁パーツ23個の生座標ではなく、部屋の幅×奥行+開口部位置に整理 |
| LLMに生データを渡して推論させる | 壁パーツ座標群をそのままLLMに渡し、居室範囲を推定させる | 旧PoC3の方式。推定は可能だが不安定。前処理で確定すべき |

## 推奨アプローチ

```
Step 1: 情報変換（3Dデータ → Claude Code認識可能形式）
├── .blend → 平面図画像（Blender正射投影レンダリング）
├── .blend → 部屋寸法JSON（bpyで壁パーツを居室単位に集約）
└── .glb → アセットBBox JSON（trimesh）

Step 2: Claude Codeによるレイアウト提案
├── 平面図画像 + 寸法JSON + アセットBBoxを入力として受け取る
├── 空間の形状・動線・開口部を理解した上で配置を判断
└── 各アセットの配置座標（position + rotation）をJSON出力

Step 3: バリデーション
├── 重なり検出（バウンディングボックス交差判定）
├── 空間内収まり検証（壁の内側判定）
├── 動線確保検証（主要経路の幅チェック）
└── Claude Code自身がバリデーションスクリプトを実行し、違反があれば修正

Step 4: Blender上での実体化と視覚的確認
├── bpyで配置座標にアセットを挿入
├── レンダリング画像をClaude Codeが確認
└── 問題があれば対話的に修正
```

**旧方式との違い:**

| | 旧方式（PoC3初回） | 新方式 |
|---|---|---|
| 入力形式 | 壁パーツ生座標JSON（23個）のみ | 平面図画像 + 整理済み寸法JSON |
| 処理エンジン | Gemini API（外部LLM） | Claude Code自身 |
| バリデーション | Pythonスクリプト（固定ロジック） | Claude Codeがスクリプト実行+判断 |
| 修正ループ | なし（1回生成のみ） | Claude Codeが視覚確認→対話的修正 |

## プロンプト設計の知見

Claude Codeに空間認識させる際のベストプラクティス（先行研究および旧PoC3の経験から）。

### 入力の構成

- **平面図画像**: 壁=黒、ドア=青、窓=緑で色分け。寸法の注釈をオーバーレイ
- **座標系の明示**: 右手系、Z-up、メートル単位をプロンプト冒頭で宣言
- **アセット情報**: 名前 + バウンディングボックス（幅×奥行×高さ）をリスト形式で提示

### 出力の構成

- JSON形式で `position(x, y, z)` + `rotation_z_degrees` + `reason`（配置理由）
- 配置理由を含めることで、Claude Codeの空間推論過程を検証可能

### 精度向上のコツ

- **段階的タスク分解**: ゾーニング → 大型家具 → 小物の順に配置
- **制約条件の明示**: 「家具同士が重なってはいけない」「ドアの前に1m以上のクリアランス」等を定量的に記述
- **LLM空間推論の限界**: 精密な座標計算は苦手。初期提案として扱い、バリデーションで補正する設計が必須

## 検証結果（2026-03-17）

> 旧方式（JSON入力のみ、Gemini API使用）での検証結果。新方式（平面図画像 + 寸法JSON、Claude Code使用）での再検証は今後実施。

### 使用モデル

Gemini 3.1 Pro（`gemini-3.1-pro-preview`）

### パイプライン

```

madori.blend                  *.glb (5個)
     │                              │

  Blender bpy → JSON            trimesh → JSON

     │                              │
     └──────────┬───────────────────┘
```

### 空間分析結果（LLMによる）

LLMはmadori.blendから抽出した壁パーツ座標群を解析し、以下を正しく推定した：

- **居室範囲**: x=[-2.20, 3.02], y=[-1.27, 2.26]（約5.2m × 3.5m）
- **ドア位置**: 南側の壁の開口部（x=0.46〜1.08の隙間）
- **窓位置**: 北側の壁（Y=4.0〜4.2付近）、バルコニー側

### 配置座標の出力結果

| アセット | 位置(x, y) | 回転 | 配置理由 |
|---------|-----------|------|---------|
| closet | (-1.93, 0.50) | 90° | 西壁に沿って配置。大型家具を壁際に寄せ圧迫感を軽減 |
| dining_table | (-0.80, 0.50) | 90° | 西寄りに配置。東側の主動線を確保 |
| desk | (2.42, 1.00) | 0° | 東壁に沿い北向き。窓からの自然光を活用 |
| chair | (2.42, 0.25) | 0° | デスクに正対。背後にスペースあり |
| counter | (-1.20, -1.07) | 0° | 南西の壁際。ドア動線を妨げない位置 |

### バリデーション結果

- 壁/柱との衝突: **なし**（23壁 + 4柱の全オブジェクトとチェック済み）
- アセット同士の重なり: **なし**（5アセットの全ペアでBBox交差判定済み）
- 主動線幅: **2.22m**（ドア→窓の経路上）
- ドアクリアランス: **3.53m**

### 考察

- 全アセットが居室内に収まり、壁・柱との衝突はゼロ
- 西側に大型家具（closet + dining_table）、東側にデスクワークエリアと機能的にゾーニングされている
- ドアから窓への主動線に家具を配置せず、幅2.2mの通路を確保できている
- LLMは壁パーツ群から居室範囲・ドア・窓の位置を正しく推論できた
- **ただしJSON入力のみでの推論であり、平面図画像の併用による精度向上は未検証**

### 課題・制約

- **入力が壁パーツ生座標のみ**: 平面図画像による視覚的な空間理解を活用できていない。新方式で改善予定
- **処理エンジンが外部API**: Gemini APIへの依存。Claude Code自身が処理エンジンとなる新方式で解消
- **アセットの実寸が手動指定**: Tripo AIのGLB出力は正規化されており軸方向も不定のため、バウンディングボックスの実寸はスクリプト内にハードコードした
- **バリデーションは2D（XY平面）のみ**: 高さ方向の衝突（例: 天井との干渉）はチェックしていない
- **修正ループなし**: 1回の生成結果のみ。Claude Code方式では対話的修正が可能になる

## 検証結果（2026-03-21）Claude Code方式

### 検証内容

Claude Codeを処理エンジンとして、空間認識→レイアウト提案→評価→修正のループを実行した。

### 構築したパイプライン

```
Phase 1: .blend → SVG（壁・柱・開口部を実座標で直接描画）
Phase 2: SVGをdrawioに埋め込み → ユーザーが部屋・設備・配置不可を手作業定義
Phase 3: drawioデータ + 壁データ → 統合SVG + room_info.json
Phase 4: アセット情報定義（assets.json: サイズ、形状、配置ルール、ゾーニング）
Phase 5: レイアウト提案（placement_plan.json → placement_engine.py → SVG）
         配置 → 評価 → 修正のrefineループ
```

### 達成できたこと

- **リアル制約のデジタル化**: .blendの3Dデータから壁・柱・開口部を実座標でSVGに変換し、Claude Codeが認識できる形にした
- **空間情報の構造化**: drawioでユーザーが部屋・配置不可領域を定義し、SVGに統合するワークフローを確立
- **家具情報の構造化**: 形状（front/back）、配置ルール、ゾーニングをassets.jsonに定義
- **配置エンジン**: Claude Codeが決めた座標で配置し、重なりチェックを行う仕組みを構築
- **refineループ**: 配置→SVG生成→評価→修正の反復サイクルを実行

### 判明した限界

**Claude Codeはリアルな生活を想像できない。**

- 座標の重なり/隙間の数値判断はできるが、「人がその空間でどう動くか」を想像できない
- カウンターの引き戸の前にダイニングテーブルを置いても「OK」と評価してしまう
- 空きスペースに家具を詰め込む思考になり、動線を阻害する配置をする
- 自己評価が甘く、問題を見逃す（ユーザー評価とのギャップが大きい）

### ビジョンとの照合

事業の軸は「リアル制約 × AI創造」。

- **リアル制約のデジタル化**: 達成。3Dデータ→SVG→構造化データの変換パイプラインが機能
- **AI創造（レイアウト提案）**: 品質不足。座標計算はできるが「現実に実装可能なレイアウト」の創造に至っていない

**核心の課題: リアル制約をどのように定義してAIに渡すとうまく認識できるか。**
これはビジョンの根幹であり、ここに注力することで独自性を発揮できる。

### 次のアプローチ（検証継続）

人間がレイアウトを考えるときの思考パターンに寄せる:

1. **画像ベースの判断**: JSON座標ではなく、SVG/画像を見て配置を判断する
2. **生活シナリオの定義**: 「朝起きて→着替えて→キッチンで朝食→デスクで仕事」のようなシナリオを定義し、そのシナリオが成立するか画像上でトレースする
3. **動線の経路表現**: 矩形の配置不可ではなく、A→B→Cの経路として動線を表現する
4. **Gemini等の画像認識の活用**: Claude Codeの空間推論の弱さを補完するため、画像認識に強いモデルの併用を検討
