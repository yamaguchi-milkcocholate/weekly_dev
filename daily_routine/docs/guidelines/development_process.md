# 開発プロセスガイドライン

## ドキュメント体系

本プロジェクトでは、以下の3種類のドキュメントを使い分ける。

| 種別                 | 配置場所         | 役割                                                                 |
| -------------------- | ---------------- | -------------------------------------------------------------------- |
| **仕様書（Spec）**   | `/docs/specs/`   | **何を作るか**を記述する。現時点の完成形ビジョンを定義する。         |
| **設計書（Design）** | `/docs/designs/` | **どうやって作るか**を記述する。サブタスクごとの実装詳細を定義する。 |
| **ADR**              | `/docs/adrs/`    | **何を採用したか**を記録する。技術選定の意思決定とその理由を残す。   |

## 開発の進め方

1. **仕様書に完成形ビジョンを記述する**
   - まず仕様書（`/docs/specs/`）に現時点で想定する完成形を記載する。
   - 仕様書は「今の理解での完成形」であり、確定版ではない。

2. **仕様書をサブタスクに分解する**
   - 仕様書の内容をもとに、実装可能な粒度のサブタスクに切り分ける。

3. **サブタスクごとに設計書を作成する**
   - 各サブタスクの実装詳細は設計書（`/docs/designs/xx_design.md`）に記載する。
   - 設計書には具体的な技術選定、データ構造、インターフェース仕様などを含む。

4. **実装とフィードバックで仕様書を更新する**
   - 実装や実装結果の検証を通じて得られた知見をもとに、仕様書を継続的にアップデートする。
   - 仕様書は何度も更新されることを前提とする。固定されたドキュメントではない。

```
仕様書（完成形ビジョン）
    ↓ 分解
サブタスク定義
    ↓ 詳細化
設計書（実装詳細）
    ↓ 実装・検証
実装結果のフィードバック
    ↓ 反映
仕様書の更新（繰り返し）
```

## ADR（Architecture Decision Records）の運用

技術選定の意思決定履歴を `/docs/adrs/` に記録する。

### いつ作成するか

- 使用する技術・ライブラリ・サービスを新たに採用する時
- 既存の技術選定を変更・撤回する時

### ファイル命名規則

```
/docs/adrs/{連番}_{選定対象の概要}.md
```

例: `001_video_generation_ai.md`, `002_bgm_generation.md`

### 記載内容

| 項目 | 内容 |
|------|------|
| タイトル | 何の技術選定か |
| ステータス | 提案 / 採用 / 撤回 |
| コンテキスト | なぜこの選定が必要になったか |
| 候補 | 検討した選択肢とそれぞれの評価 |
| 決定 | 何を採用し、なぜその判断に至ったか |
| 結果 | 採用による影響・トレードオフ |
